# [보고서] 개별 페이지 스크랩 방식 진단 및 효율화 방향

> **작성일**: 2026-03-10
> **작성자**: Gemini CLI
> **대상**: scrap_sns 프로젝트 내 X(Twitter), Threads, LinkedIn 스크래퍼

## 1. 개요
현재 프로젝트의 SNS별 개별 페이지 상세 수집(Consumer) 방식을 진단하고, 사용자 제안인 **Playwright CLI** 도입과 기존 **Python-based Playwright** 방식의 효율성을 비교 분석합니다.

---

## 2. 비교 분석 (Playwright CLI vs Python Script)

| 항목 | Python-based Playwright (AS-IS) | Playwright CLI (Alternative) | 비고 |
| :--- | :---: | :---: | :--- |
| **로그인 세션 유지** | ✅ ██████ (매우 용이) | ❌ ░░░░░░ (매번 재로그인 필요) | `persistent_context` 필수 |
| **비동기 병렬 처리** | ✅ ██████ (탭 3개 이상 병렬) | ❌ ░░░░░░ (순차적 실행) | `asyncio` + `Semaphore` 활용 |
| **네트워크 가로채기** | ✅ 지원 (`on("response")`) | ❌ 미지원 (DOM 스냅샷 전용) | GraphQL 데이터 수집 핵심 |
| **안티봇 우회 능력** | ✅ ████░░ (고급 설정 가능) | ⚠️ ░░░░░░ (기본 설정 제한적) | `--disable-blink-features` 등 |
| **유지보수 효율** | ✅ ████░░ (Python 코드 통합) | ❌ ░░░░░░ (셸 명령어 분절) | 데이터 스키마 정합성 관리 |
| **구현 속도(신규)** | ██░░░░ (코드 작성 필요) | ✅ ██████ (한 줄로 즉시 실행) | 간단한 스냅샷 시에는 유리 |

## 3. 상세 진단 결과

### ❌ Playwright CLI 도입 "부적합" 판정
1. **세션 유지 한계**: SNS 스크래핑은 로그인 세션 유지가 핵심입니다. CLI는 매번 브라우저를 초기화하므로, 매 세션마다 로그인을 요구하게 되어 자동화가 불가능합니다.
2. **기술적 제약**: 현재 LinkedIn 및 Threads 스크래퍼의 핵심인 **네트워크 응답 가로채기(Response Interception)** 기능이 CLI에서는 지원되지 않습니다.
3. **성능 저하**: CLI는 한 번에 하나의 URL만 처리하므로, 현재 Threads 스크래퍼가 구현 중인 3개 이상의 탭을 동시에 활용하는 병렬 처리가 불가능하여 수집 속도가 현저히 떨어집니다.

### ✅ 기존 방식의 문제점 및 개선점
기존 방식은 "기술적 한계"보다는 **"구조적 부채"**가 더 큽니다.
- **중복 코드**: `reorder_post()`, `clean_text()` 등 핵심 유틸리티가 파일마다 중복 구현되어 유지보수 비용이 높음.
- **비효율적 실행**: X(Twitter) 스크래퍼가 여전히 동기(Sync) 방식으로 순차 처리되어 잠재적 속도 향상 기회를 놓치고 있음.
