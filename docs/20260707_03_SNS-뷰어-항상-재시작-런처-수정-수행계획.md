# SNS Viewer Always-Restart Launcher Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `wscript sns_hub.vbs` 또는 `npm run view` 실행 시 기존 뷰어 서버가 정상 응답 중이어도 항상 종료 후 새 코드로 재시작한다.

**Architecture:** 서버 실행 파일명을 `server.py`에서 `scrap_sns_server.py`로 바꿔 프로세스 command line 식별성을 높인다. `scripts/restart_viewer_server.ps1`은 `/api/status`나 코드 신선도 검사로 재시작을 건너뛰지 않고, 5000번 포트 점유 프로세스 command line에 `scrap_sns_server.py`가 있을 때만 종료한다. 디렉터리 일치, 절대경로 일치, 수정시각 비교는 재시작 판단에 넣지 않는다.

**Tech Stack:** Python Flask, PowerShell, VBScript, Batch, pytest, Playwright.

---

## 결정 사항

- 런처 실행은 항상 재시작이다. 이미 응답 중인 서버라도 재사용하지 않는다.
- 프로세스 종료 guard는 `scrap_sns_server.py` 파일명 포함 여부만 사용한다.
- 현재 `$ProjectRoot`와 기존 프로세스의 작업 디렉터리 또는 절대경로 일치 여부는 확인하지 않는다.
- 5000번 포트를 다른 프로세스가 사용 중이면 종료하지 않고 실패한다.
- `/api/status`, `/api/get-tag-catalog`는 재시작 생략 판단이 아니라 새 서버 기동 후 readiness 확인에만 사용한다.
- 저장 데이터 형식은 바꾸지 않는다. `output_total`, `web_viewer/sns_tags.json`, `web_viewer/sns_user_metadata.json` 재처리는 필요 없다.

### 5000번 포트 전환 메모

구현 직후 5000번 포트는 rename 전 프로세스가 점유 중이었다.

```markdown
PID 22256
CommandLine: "C:\Users\ahnbu\AppData\Local\Microsoft\WindowsApps\python.exe" server.py
CreationDate: 2026-07-07 13:21:06 KST
```

새 `restart_viewer_server.ps1`를 5000번에 실행하면 아래처럼 실패한다.

```markdown
Port 5000 is used by PID 22256, but it does not look like scrap_sns_server.py. CommandLine: "C:\Users\ahnbu\AppData\Local\Microsoft\WindowsApps\python.exe" server.py
```

이는 새 guard가 의도대로 동작한 결과다. `server.py`는 이제 식별 대상이 아니므로 자동 종료하지 않는다. 실제 5000번을 새 서버로 전환하려면 현재 떠 있는 legacy `server.py` 프로세스를 한 번 종료한 뒤 `wscript sns_hub.vbs` 또는 `npm run view`를 실행해야 한다.
