# LinkedIn 인증 접근 방식 테스트 계획과 결과

## 발단: 사용자 요청

LinkedIn 수집이 `authwall/checkpoint`로 막히는 상황에서, `scrap_sns` 수집기와 인증갱신기의 Playwright 접근 환경이 서로 다른 점이 확인됐다.

사용자 요청은 다음 두 가지다.

- 현재 방식이 아닌 접근 방식으로 일단 LinkedIn 접근이 되는지 테스트한다.
- 테스트 계획을 문서로 저장하고, 테스트 결과를 같은 문서에 기록한다.

## 작업 상세내역

현재 확인된 구조는 다음과 같다.

| 구분 | scrap_sns LinkedIn 수집기 | scrap_sns 인증갱신기 | sns_insight_update LinkedIn 수집기 |
|---|---|---|---|
| 핵심 파일 | `linkedin_scrap.py` | `C:\Users\ahnbu\.config\auth\renew.py` | `collectors/linkedin.py` |
| 인증 파일 | `C:\Users\ahnbu\.config\auth\linkedin\storage_state.json` | 동일 파일을 갱신 | 동일 파일을 사용 |
| Playwright 방식 | `chromium.launch` + `new_context(storage_state)` | `chromium.launch` + `new_context(storage_state)` | `chromium.launch` + `new_context(storage_state)` |
| user_agent | `Chrome/122` 명시 | 명시 없음 | `Chrome/122` 명시 |
| locale | `ko-KR` | 명시 없음 | `ko-KR` |
| viewport | `900x500` | 기본값 또는 `--web`에서 `1100x800` | `1280x1000` |
| 대상 URL | `my-items/saved-posts` | `linkedin.com/login` | `recent-activity/all` |

이번 테스트는 운영 코드를 바로 수정하지 않고, 기존 인증 파일을 읽는 비파괴 probe로 접근 환경만 바꿔 비교한다.

## 의사결정 기록

- 결정: 운영 코드 수정 전, 접근 방식별 probe 결과를 먼저 확보한다.
- 근거: 통과하는 방식이 확인되어야 `renew.py` 또는 인증 구조 개선의 방향을 정할 수 있다.
- 트레이드오프: 기존 `storage_state.json` 자체가 이미 LinkedIn에서 차단된 상태라면, user_agent/locale만 바꾼 probe는 모두 실패할 수 있다. 이 경우 다음 단계는 수동 로그인 포함 테스트 또는 persistent profile 테스트가 된다.

### 추가 테스트 결정: 로그인 카드 클릭

사용자가 관찰한 LinkedIn 로그인 화면에는 `로그인중 / 이 페이지에 남아 있을 경우 로그인됩니다` 문구와 계정 카드가 잠깐 표시된다. 로그인 자동 진행이 실패하거나 중간 화면에서 멈추는 경우, 이 계정 카드/로그인 요소를 fallback으로 클릭하면 성공확률이 올라가는지 가설 검토한다.

이 테스트는 기존 테스트 매트릭스에서 `storage_state` 재사용 probe와 새 인증 생성 테스트 사이에 둔다. 이유는 다음과 같다.

- 기존 인증 파일을 그대로 쓰므로 "새 인증 생성"은 아니다.
- 단순 URL 접근이 아니라 LinkedIn 로그인 중간 UI를 조작하므로 기존 비파괴 probe와는 다른 가설이다.
- 성공하면 `renew.py`에 "로그인 중간 UI 감지 후 계정 카드/로그인 요소 클릭" fallback 단계를 넣는 작은 개선 후보가 된다.
- 실패하면 `storage_state` 재사용 계열은 접고, 새 인증 생성 또는 persistent profile 검증으로 넘어가야 한다.

안전 기준:

- canonical `C:\Users\ahnbu\.config\auth\linkedin\storage_state.json`은 덮어쓰지 않는다.
- 성공 시에도 임시 storage 파일에만 저장한다.
- 보안 챌린지, 비밀번호, 2FA, CAPTCHA 입력은 자동화하지 않는다.
- 클릭 대상은 사용자가 관찰한 계정 카드 또는 `로그인중` 화면의 로그인 진행 요소로 한정한다.

### 추가 범위 결정: 수동 export 제외

2026-05-05 11:20 KST 기준, 사용자는 수동 export 기반 수집 MVP를 현재 범위에서 제외한다고 결정했다.

따라서 다음 검토 범위는 일반 Chrome 프로필 기반 수집 가능성으로만 좁힌다. HTML/JSON/클립보드 복사, 브라우저 확장, 사용자가 saved-posts 화면에서 데이터를 넘겨주는 수동 export MVP는 이번 테스트와 설계 범위에 포함하지 않는다.

- 진짜 원인은 갱신기와 수집기의 fingerprint 불일치이며, 갱신기 환경(`renew_storage_state()` 본체)을 수집기와 정렬하면 saved-posts가 통과된다. 다음 단계는 옵션 A로 `scripts/auth_runtime/renew.py:renew_storage_state()` 본체에 `user_agent`(Chrome/122) + `locale`(ko-KR) + `viewport`(1280x1000)를 추가하는 코드 변경 plan(`docs/plans/20260505_03_renew.py-환경정렬-LinkedIn-fingerprint-회복.md`)이다.

### 추가 관찰

첫 공식 수집기 실행은 `UnicodeEncodeError: cp949`로 인증 판정 전에 중단됐다. `PYTHONIOENCODING=utf-8`을 지정하자 실제 인증 판정까지 진행됐고, `SNS_AUTH_REQUIRED`가 출력됐다.

화면 좌표 문제도 확인됐다. `linkedin_scrap.py`와 초기 진단 스크립트는 창 위치를 `5000,200`으로 잡고 있었지만, 현재 모니터는 `0,0,1920x1200` 단일 화면이다. 이후 임시 테스트는 `80,80` 또는 `120,120`으로 조정해 보이는 위치에서 실행했다.

## 리스크 및 미해결 이슈

- LinkedIn은 같은 쿠키라도 시점, IP, 브라우저 fingerprint, checkpoint 상태에 따라 결과가 달라질 수 있다.
- 기존 인증 파일은 이미 서버 쪽에서 재검증 대상으로 표시된 것으로 보인다.
- 이번 테스트는 새 `storage_state` 생성과 Playwright 기반 persistent profile 생성까지 검증했다. 둘 다 "로그인된 것처럼 보이는 상태"와 "saved-posts 접근 가능 상태"가 다를 수 있음을 보였다.
- 일반 Chrome 기본 프로필은 saved-posts에 접근되지만, 이 사실만으로 Playwright가 같은 프로필을 안정적으로 재사용할 수 있다고 단정할 수는 없다. 기본 Chrome 프로필은 실행 중 잠금, 개인정보/동기화 상태, LinkedIn 자동화 감지 리스크를 함께 검토해야 한다.
- 현재 열린 일반 Chrome에 CDP로 붙을 수 없고, 기본 Chrome 프로필을 Playwright가 직접 여는 방식도 실패했다. 따라서 기본 프로필 기반 자동 수집은 "그대로 적용"하기 어렵다.
- 실제 갱신 성공 여부는 최종적으로 `linkedin_scrap.py --mode update` 또는 LinkedIn preflight 통과로 확인해야 한다.
- saved-posts와 recent-activity가 서로 다른 인증 판정을 보인다. 따라서 LinkedIn 인증 성공 기준은 recent-activity 접근만으로 잡으면 안 된다.
