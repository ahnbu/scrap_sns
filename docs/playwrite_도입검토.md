# Threads Crawler 로직 비교 및 분석 보고서

본 문서는 스레드 크롤러의 수집 안정성을 높이기 위해 적용된 업데이트 로직과 텍스트 필터링 로직에 대한 상세 분석 내용을 담고 있습니다.

## 1. 업데이트 로직 비교 (Stop Sign 전략)

기존의 단일 기준점 방식에서 발생하던 문제를 해결하기 위해 '징검다리(Multiple Stop Codes) 전략'을 도입했습니다.

| 항목            | 기존 로직 (Single Stop Code)           | 개선 로직 (Multiple Stop Codes)            |
| :-------------- | :------------------------------------- | :----------------------------------------- |
| **기준점 개수** | 최신 게시물 **1개**                    | 최신 게시물 **5개** (최신순 정렬)          |
| **중단 조건**   | 기준 게시물과 정확히 일치할 때         | 5개 중 **어느 하나라도** 일치할 때         |
| **장점**        | 로직이 단순하고 명확함                 | 기준 게시물이 삭제되어도 업데이트가 중단됨 |
| **단점**        | 기준 게시물 삭제 시 끝까지 스크롤 발생 | 없음 (보유 데이터 정합성 유지)             |

### 도입 배경

- 사용자의 "저장됨" 목록에서 최상단 게시물이 삭제되거나 작성자가 글을 내릴 경우, 크롤러가 멈춰야 할 표지판을 찾지 못하고 바닥까지 스크롤하는 현상이 발견됨.
- 이를 방지하기 위해 상위 5개의 게시물을 모두 '정지 신호' 후보로 등록하여, 하나가 사라지더라도 다음 글에서 안전하게 멈추도록 개선함.

## 2. 텍스트 필터링 로직 분석 (링크드인 삭제 사례)

터미널 로그와 실제 저장 데이터 간의 텍스트 차이(본문 시작 단어 누락)에 대한 분석 결과입니다.

### 현상 보고

- 게시물 본문이 `"링크드인\n1일\n오랫동안..."`으로 시작할 때, 로그에는 `"오랫동안..."`만 표시되는 현상 발생.

### 원인 분석

- **Greedy 필터링 로직**: 현재 `clean_text` 함수는 실제 본문이 시작되었다고 판단할 때까지 상단의 모든 줄을 메타데이터(헤더)로 간주하고 무시함.
- **오인 기전**:
  1. "링크드인" (첫 줄): 유저명과 다르지만 본문 시작 확신이 없어 임시 대기.
  2. "1일" (둘째 줄): 날짜 패턴(`\d+일`)에 해당하여 삭제.
  3. "오랫동안" (셋째 줄): 날짜 패턴이 아니므로 여기서부터 본문 시작(`is_body_started=True`) 판단.
- 이 과정에서 본문 시작 전의 "링크드인"이 의도치 않게 메타데이터의 일부로 취급되어 삭제됨.

### 최종 결정

- **현행 유지**: 필터링 로직을 완화(Strict 방식)할 경우, 오히려 더 많은 불필요한 메타데이터(수정됨, AI Threads 등)가 본문에 섞여 들어올 위험이 있음.
- **결론**: 데이터의 정밀도를 위해 현재의 "공격적인 필터링" 방식을 유지하며, 로그와 저장 데이터의 일관성 문제는 '과거 데이터 병합' 과정에서 발생하는 자연스러운 현상임을 확인함.

## 3. 디버깅 강화 사항

- 수집되는 모든 게시물의 로그에 `[Code ID]`를 병기하여, 어떤 ID의 게시물이 수집되고 있는지 또는 중복 제거되고 있는지 터미널에서 즉각 확인 가능하도록 개선함.
- **출력 예시**: `+ [Network] [DT7_LsBj_kt] 오랫동안 셀프 브랜딩을...`
