# PDCA 보고서: '최초 수집 시점' 기반 저장순 정렬 구현 완료

## 1. 개요 (Act)
- **목표**: 모든 플랫폼 수집기에 '최초 인지 시점(`crawled_at`) 기록' 로직을 도입하고, 통합 병합 시 이 시점을 기준으로 정렬하여 완벽한 '저장순' 피드를 구현함.
- **결과**: Twitter, Threads, LinkedIn 모든 스크래퍼가 동일한 메타데이터 보존 전략을 따르게 되었으며, 통합 피드에서 정렬 안정성이 확보됨.

## 2. 주요 변경 사항

### A. 모든 스크래퍼 공통 (`twitter_scrap.py`, `thread_scrap.py`, `linkedin_scrap.py`)
- **`crawled_at` 도입**: 신규 게시물 발견 시 ISO 8601(ms 포함) 형식으로 최초 수집 시점 기록.
- **메타데이터 보존**: `all` 모드 또는 `update` 모드 실행 시 기존 데이터의 `crawled_at`과 `sequence_id`를 덮어쓰지 않고 유지함.
- **`max_sequence_id` 관리**: 각 플랫폼 파일의 메타데이터에 최대 ID를 기록하여 연속성 보장.

### B. 통합 수집기 (`total_scrap.py`)
- **Traceability**: 병합 시 플랫폼별 원래 ID를 `platform_sequence_id` 필드에 보관.
- **안정적 정렬 (Stable Sort)**: `(crawled_at, platform_sequence_id)` 복합 키를 사용하여 정렬 기준 확립. 동시 수집된 게시물도 플랫폼 내 수집 순서대로 나열됨.
- **전역 ID 재부여**: 정렬된 결과에 대해 최종 `sequence_id`를 부여하여 웹 뷰어의 정렬 소스로 활용.

## 3. 검증 결과
- **정렬 확인**: `output_total/total_full_20260212.json` 내 데이터가 수집 시점순으로 정확히 정렬됨을 확인.
- **데이터 무결성**: 기존 게시물의 수집 시점이 `all` 모드 실행 후에도 변하지 않고 유지됨을 확인.

## 4. 향후 과제
- `crawled_at` 정보가 없는 완전한 레거시 데이터에 대한 보정 로직 (현재는 `timestamp`로 대체 중)이 필요할 경우 추가 작업 가능.
- 웹 뷰어 UI에서 '저장순' 레이블을 '수집순'으로 더 명확히 표기하는 방안 검토.
