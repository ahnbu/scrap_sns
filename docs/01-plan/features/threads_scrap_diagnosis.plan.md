# PDCA 계획: Threads 스크래퍼 수집 중단 문제 진단 및 해결

## 1. 개요 (Overview)
- **대상**: `thread_scrap.py`
- **문제**: Threads "저장됨" 페이지에서 1건의 게시물만 수집되고, 스크롤을 해도 네트워크를 통한 추가 수집이 이루어지지 않음.
- **목표**: 네트워크 응답 구조를 재분석하여 데이터 수집 로직을 복구하고 정상적으로 증분/전체 수집이 가능하도록 함.

## 2. 현재 상황 및 가설 (Current Status & Hypotheses)
### 2.1. 현상 분석
- 로그상 자동 로그인은 성공하며, 첫 화면의 DOM 스캔으로 1건을 수집함.
- 이후 스크롤을 5회 수행하지만 `handle_response`에서 유의미한 데이터를 0건 수집함.
- `🏁 더 이상 새로운 데이터가 없습니다.` 메시지와 함께 종료됨.

### 2.2. 원인 가설
- **가설 1: 응답 구조 변경** - Threads의 GraphQL 응답 스키마가 변경되어 기존의 `xdt_text_app_viewer` 또는 `text_post_app_user_saved_posts` 경로가 더 이상 유효하지 않음.
- **가설 2: 네트워크 캡처 누락** - 특정 조건(예: 응답 크기, 타이밍)으로 인해 Playwright가 관련 응답을 놓치거나 처리하지 못함.
- **가설 3: 필터링 로직 과작동** - `process_network_post` 내부의 작성자 확인(`current_user_pk != root_user_pk`) 등 필터링 로직이 잘못되어 데이터를 버리고 있을 가능성.

## 3. 해결 방향 (Proposed Actions)
- **진단**: `handle_response` 내부에서 발생하는 예외를 로깅하고, 실제 수신되는 JSON 응답 구조를 파일로 저장하여 분석.
- **수정**: 분석된 새로운 데이터 구조에 맞춰 추출 경로 업데이트.
- **강화**: DOM 기반 수집 로직을 보강하여 네트워크 캡처 실패 시에도 최소한의 데이터를 확보할 수 있도록 개선.
