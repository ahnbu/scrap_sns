# PDCA 계획: Twitter 목록 수집기 스크롤 누락 및 조기 종료 문제 해결

## 1. 개요 (Overview)
- **대상**: `twitter_scrap.py`
- **문제**: 북마크 수집 시 초기 로딩된 20개 항목만 가져오고, 스크롤을 해도 추가 항목(최소 34개 예상)을 수집하지 못한 채 "새로운 게시물이 없습니다"라며 조기 종료됨.
- **목표**: 스크롤 시 새로운 데이터를 정상적으로 감지하여 전체 북마크 목록을 누락 없이 수집함.

## 2. 현재 상황 및 가설 (Current Status & Hypotheses)
### 2.1. 현상 분석
- 로그상 `handle_response`(Network)를 통해 데이터가 수집되지만, `consecutive_no_new` 카운터가 5에 도달하여 스크롤이 중단됨.
- 수집된 개수가 정확히 20개에서 멈추는 것은 Twitter의 첫 번째 페이지 로딩 단위와 일치함.

### 2.2. 원인 가설
- **가설 1: Network-DOM 로직 충돌 (Race Condition)**
    - `handle_response`에서 네트워크 응답을 먼저 받아 `all_posts_map`에 저장하면, 뒤이어 실행되는 `extract_from_html`(DOM) 단계에서는 해당 데이터가 이미 맵에 존재하므로 `round_new`가 0이 됨.
    - 이로 인해 `consecutive_no_new`가 증가하여 실제로는 새로운 데이터가 들어왔음에도 불구하고 "새로운 데이터 없음"으로 판단하고 종료함.
- **가설 2: 스크롤 속도 및 타이밍 이슈**
    - 스크롤 후 대기 시간(2.5초)이 부족하여 네트워크 응답이 오기 전에 다음 루프가 실행되거나, DOM에 요소가 렌더링되지 않았을 가능성.
- **가설 3: 중단점(`stop_ids`) 오작동**
    - `update` 모드에서 기존 데이터와의 비교 로직이 잘못되어 의도치 않게 조기 중단됨. (하지만 현재 로그는 신규 수집 상태이므로 가능성 낮음)

## 3. 해결 방향 (Proposed Actions)
- **로직 통합**: Network 수집 여부와 DOM 수집 여부를 합산하여 "새로운 데이터 발견 여부"를 판단하도록 개선.
- **카운터 최적화**: `handle_response` 내에서 새로운 데이터가 추가되었을 때도 `consecutive_no_new`를 초기화하도록 변경.
- **안정성 강화**: 스크롤 대기 시간을 조정하고, 스크롤 후 페이지 하단 도달 여부를 보다 정확히 체크.
