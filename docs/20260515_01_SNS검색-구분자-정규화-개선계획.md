# SNS 검색 구분자 정규화 개선계획

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. 현재 프로젝트 규칙에 따라 별도 worktree를 자동 생성하지 않고 현재 브랜치에서 직접 작업한다. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** SNS 허브 검색에서 사용자가 `slide grab` 또는 `slides grab`처럼 자연스럽게 입력해도 본문에 있는 `slides-grab` 게시물이 검색되도록 한다.

**Architecture:** 기존 `/api/search`와 `utils/query-sns.mjs search`의 exact substring 검색을 유지하되, 검색어와 검색 대상 모두에 최소 구분자 정규화를 적용한다. FTS, fuzzy, entity dictionary, compact full-text 인덱스는 이번 범위에서 제외한다.

**Tech Stack:** Python Flask `server.py`, Node.js ESM `utils/query-sns.mjs`, pytest, node:test.

## 검수 보완 요약

계획 검수에서 크리티컬한 방향 오류는 발견하지 못했다. 다만 후속 구현자가 그대로 복사했을 때 생길 수 있는 실행 리스크 3가지를 보완했다.

| 보완 항목 | 문제 | 보완 |
|---|---|---|
| Flask 테스트 query 전달 | 수동 `%20` 인코딩은 쿼리 문자가 늘 때 깨지기 쉬움 | `query_string={...}` 사용으로 변경 |
| Python import 위치 | `import unicodedata`를 helper 근처에 두면 파일 import 스타일과 어긋남 | top-level import에 추가하도록 명시 |
| CLI import-safe 조건 | `pathToFileURL(process.argv[1])` 비교는 Windows 상대경로·정규화에서 헷갈릴 수 있음 | `fileURLToPath(import.meta.url) === path.resolve(process.argv[1])`로 변경 |

## 배경

사용자가 SNS 허브에서 `slide grap`으로 검색했을 때 `slides-grab` 게시물이 나오지 않는 이유를 조사했다. 최신 통합 데이터에는 `slides-grab` 포함 게시물이 실제로 있었고, `total_full_20260515.json` 기준 `slides-grab`은 5건이었다.

초기 원인은 `/api/search`가 검색어 전체를 그대로 lower-case 처리한 뒤 `_searchable` 문자열에 포함되는지만 보는 구조였기 때문이다. 즉 현재 동작은 아래 비교에 가깝다.

```python
"slide grab" in "slides-grab"
```

이 비교는 공백과 하이픈이 다르고, `slide`와 `slides`가 한 덩어리 문자열로 비교되기 때문에 실패한다.

논의 과정에서 `grap` ↔ `grab` 같은 오타 보정까지 검토했지만, 실제 목적은 오타 보정이 아니라 `slide grab`으로 `slides-grab`을 찾는 것이다. 따라서 이 계획은 fuzzy 검색을 넣지 않고, 구분자와 단어 분리 문제만 해결한다.

## 현행 사실

| 항목 | 확인 결과 | 판단 |
|---|---|---|
| 데이터 존재 | ✅ `slides-grab` 포함 게시물 5건 존재 | 데이터 누락 아님 |
| 서버 검색 | ❌ `query in _searchable` exact substring | 사용자 기대보다 좁음 |
| CLI 검색 | ❌ `String(...).toLowerCase().includes(needle)` | 서버와 같은 한계 |
| 뷰어 UI | ✅ `/api/search`를 200ms debounce로 호출 | UI 변경 불필요 |
| 영구 데이터 | ✅ 검색 캐시는 프로세스 메모리 | JSON 재처리 불필요 |

## 목표 동작

| 검색어 | `slides-grab` 매칭 | 이유 |
|---|---|---|
| `slides-grab` | ✅ | 기존 exact 의도 유지 |
| `slides grab` | ✅ | `-`를 공백처럼 처리 |
| `slide grab` | ✅ | `slide`와 `grab`이 모두 검색 대상에 포함 |
| `slide-grab` | ✅ | 검색어 쪽 `-`도 공백처럼 처리 |
| `slide grap` | ❌ | `grab` 오타 보정은 이번 범위가 아님 |
| `slidesgrab` | ❌ | 붙여쓰기 compact 검색은 이번 범위가 아님 |

주의: `slide grap`은 `slides-grab`을 오타 보정으로 찾지는 않는다. 다만 실제 본문에 `grap`이 들어간 `graphics`, `paragraphs`, `geography` 등이 있으면 별도 결과가 나올 수 있다. 이것은 fuzzy가 아니라 literal 부분 일치의 결과다.

## 검토한 옵션

| 옵션 | 효과 | 비용/위험 | 결정 |
|---|---|---|---|
| 현행 exact substring 유지 | ✅ 구현 비용 없음 | ❌ `slide grab` → `slides-grab` 누락 | 제외 |
| 구분자 정규화 + token AND | ✅ 목적 달성, 범위 작음 | ⚠️ 다단어 검색 결과가 조금 넓어짐 | 채택 |
| SQLite FTS5 `trigram` | ✅ substring 검색 성능·품질 개선 | ⚠️ 현재 규모에서는 과하고 인덱스 설계 필요 | 제외 |
| full-body fuzzy | ✅ 오타 보정 가능 | ❌ `grap`이 `graphics` 등으로 오검색 확산 | 제외 |
| hyphenated entity dictionary | ✅ 도구명 오타 보정에 적합 | ⚠️ 별도 추출·랭킹 설계 필요 | 후속 후보 |
| compact full-text 저장 | ⚠️ `slidesgrab` 대응 가능 | ❌ 본문 검색 문자열 추가 증식 | 제외 |

## 설계 원칙

- 검색어와 검색 대상의 `-`, `_`만 공백으로 정규화한다.
- 모든 문장부호를 제거하지 않는다. `C++`, URL, `@user` 같은 검색을 불필요하게 깨지 않기 위해서다.
- 다단어 검색은 정규화된 term이 모두 포함될 때 매칭한다.
- `slide`가 `slides` 안에 포함되는 수준의 부분 일치는 허용한다.
- 오타 보정은 하지 않는다.
- 정렬, 플랫폼 필터, limit, offset, gzip, ETag 정책은 변경하지 않는다.
- `output_total/*.json`, 태그 JSON, 브라우저 `localStorage`는 변경하지 않는다.

```python
_SEARCH_SEPARATOR_RE = re.compile(r"[-_]+")
_SEARCH_WHITESPACE_RE = re.compile(r"\s+")


def _normalize_search_text(value):
    normalized = unicodedata.normalize("NFKC", str(value or "")).casefold()
    normalized = _SEARCH_SEPARATOR_RE.sub(" ", normalized)
    return _SEARCH_WHITESPACE_RE.sub(" ", normalized).strip()


def _split_search_terms(value):
    return [term for term in _normalize_search_text(value).split(" ") if term]


def _matches_search_query(searchable, query):
    normalized_query = _normalize_search_text(query)
    if not normalized_query:
        return False
    if normalized_query in str(searchable or ""):
        return True
    terms = _split_search_terms(query)
    return bool(terms) and all(term in str(searchable or "") for term in terms)
```

```javascript
function normalizeSearchText(value = '') {
  return String(value || '')
    .normalize('NFKC')
    .toLowerCase()
    .replace(/[-_]+/g, ' ')
    .replace(/\s+/g, ' ')
    .trim();
}

function splitSearchTerms(value = '') {
  return normalizeSearchText(value).split(' ').filter(Boolean);
}

function matchesSearchText(value, query) {
  const haystack = normalizeSearchText(value);
  const normalizedQuery = normalizeSearchText(query);
  if (!normalizedQuery) return false;
  if (haystack.includes(normalizedQuery)) return true;
  const terms = splitSearchTerms(query);
  return terms.length > 0 && terms.every((term) => haystack.includes(term));
}
```

## 부록 A. 이 계획이 도출된 논의 맥락

1. 최초 질문은 `slide grap` 검색이 `slides-grab` 게시물을 찾지 못하는 이유였다.
2. 조사 결과 데이터에는 `slides-grab` 게시물이 있었고, 문제는 검색 방식이었다.
3. 현재 검색은 검색어 전체가 본문에 연속 문자열로 들어 있어야 하는 exact substring 구조였다.
4. fuzzy를 검토했지만 실제 목적은 오타 보정이 아니라 `slide grab`으로 `slides-grab`을 찾는 것이었다.
5. 실제 데이터에서 `grap`은 `graphics`, `paragraphs`, `geography` 같은 단어를 끌어올릴 수 있어 full-body fuzzy는 위험하다고 판단했다.
6. 결론적으로 구분자 정규화와 다단어 AND 부분일치만 적용하는 최소 변경이 적정선이다.

## 부록 D. 외부 리서치 출처

- [SQLite FTS5 Extension](https://www.sqlite.org/fts5.html) — `trigram` tokenizer는 3글자 단위 substring 검색에 적합하나, 이번 범위에는 과함.
- [PostgreSQL pg_trgm](https://www.postgresql.org/docs/current/pgtrgm.html) — trigram similarity는 빠른 유사도 검색에 유용하지만 threshold/ranking 설계가 필요함.
- [Algolia typoTolerance](https://www.algolia.com/doc/api-reference/api-parameters/typoTolerance/) — typo tolerance는 ranking 기준과 결합되어야 하며 무조건 결과에 섞는 기능이 아님.
- [Algolia Search in hyphenated attributes](https://www.algolia.com/doc/guides/managing-results/optimize-search-results/typo-tolerance/how-to/how-to-search-in-hyphenated-attributes/) — 표시용 값과 검색용 포맷을 분리하는 패턴을 확인.
- [Meilisearch typo tolerance](https://www.meilisearch.com/docs/resources/internals/typo_tolerance) — typo 허용은 단어 길이와 typo 수 기준을 둔다.
- [Elasticsearch fuzzy query](https://www.elastic.co/docs/reference/query-languages/query-dsl/query-dsl-fuzzy-query/) — fuzzy는 edit distance expansion이며 성능 비용과 확장 수 제한이 필요함.
