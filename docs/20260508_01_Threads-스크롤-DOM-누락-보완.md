# Threads 스크롤 DOM 누락 보완 실행 계획

## 발단: 사용자 요청

사용자가 Threads 저장글 `https://www.threads.com/@ee.yxx/post/DYBIicQkumF` 누락 여부 확인과 원인 분석을 요청함.

조사 결과 대상 글은 permalink 직접 접근과 parser 추출은 가능하지만, 최신 `output_threads` 및 `output_total` 산출물에는 존재하지 않음. 이후 사용자가 HTML bootstrap 파싱 추가는 변경범위가 크며, 기존 DOM 기반 보완이 더 적합한지 검토를 요청함.

## 작업 상세내역

초기 조사에서 `DYBIicQkumF`는 `output_threads`, `output_total`, `scrap_failures_threads.json` 및 `utils/query-sns.mjs get DYBIicQkumF` 조회에서 모두 미발견.

permalink 직접 조회는 성공. `utils.threads_parser` 기준 `platform_id=DYBIicQkumF`, `username=ee.yxx`, `created_at=2026-05-07 09:42:00` 추출 가능.

추가 대조 결과, 현재 `/saved` 초기 HTML에는 21개 post code가 존재했고 이 중 13개가 최신 로컬 simple 산출물에 없음. 대상 글은 초기 HTML code 목록의 6번째이며, 초기 DOM anchor에는 없으나 두 번째 스크롤 후 DOM에 `/@ee.yxx/post/DYBIicQkumF` 링크로 렌더링됨.

따라서 원인은 상세 consumer나 병합 단계가 아니라 producer의 수집 공백. 현재 producer는 초기 DOM을 1회만 스캔하고, 이후 스크롤 단계에서는 네트워크 응답만 처리함. Threads가 초기 HTML에 preload한 글을 스크롤 후 DOM으로 렌더링하지만 새 네트워크 응답이 없을 수 있어 누락 발생.

## 의사결정 기록

| 선택지 | 변경 범위 | 안정성 | 성능 영향 | 판정 |
|--------|-----------|--------|-----------|------|
| HTML bootstrap 파싱 추가 | 큼 | ⚠️ Threads 내부 payload 변화에 민감 | 빠름 | 보류 |
| 스크롤 후 DOM 재스캔 | 작음 | ✅ 기존 DOM 수집 방식 유지 | ⚠️ href 선검사로 제한 필요 | 채택 |
| 현상 유지 | 없음 | ❌ 누락 반복 가능 | 빠름 | 기각 |

> 정렬 기준: 변경 리스크와 현재 누락 원인 해소 가능성을 우선함.

- 결정: 네트워크 응답 수집을 주력으로 유지하고, 스크롤 후 DOM 재스캔을 누락 보완 안전망으로 추가.
- 근거: 대상 글은 실제로 스크롤 후 DOM에 나타나므로 HTML parser 신설 없이 기존 DOM 수집 철학으로 보완 가능.
- 트레이드오프: 스크롤마다 DOM 조회 비용이 추가되지만, 먼저 post href/code만 확인하고 신규 code에 대해서만 본문·미디어 추출을 수행하여 속도 저하를 제한.

## 실행계획 원문

아래 내용은 linked plan 파일이 아니라 실행 전 TodoList/update_plan을 보존한 것이다.

1. `thread_scrap.py`의 초기 DOM 추출 로직을 재사용 가능한 함수로 분리.
2. 스크롤 루프에서 네트워크 응답 처리 후 DOM href/code 재스캔 추가.
3. DOM 재스캔에서 신규 code가 발견되면 기존 필드 형식으로 보완 수집하고 `no_new_data_count`를 리셋.
4. DOM 재스캔 동작을 unit test로 고정.
5. 관련 테스트와 대상 URL 조회로 검증.

## 리스크 및 미해결 이슈

- 실제 Threads `/saved`의 렌더링 상태와 로그인 세션 상태에 따라 대상 글이 재수집 시점에 노출되지 않을 수 있음.
