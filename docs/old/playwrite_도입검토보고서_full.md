# ⚔️ 비교 분석: `sns_crawler` vs `remorses/playwriter` MCP

사용자가 요청한 `remorses/playwriter` MCP를 활용한 새로운 방식과 기존 `sns_crawler` 방식의 비교 분석 결과입니다.

## 1. 개요 요약

| 구분            | **기존 방식 (`sns_crawler`)**                | **제안 방식 (`remorses/playwriter` MCP)** |
| :-------------- | :------------------------------------------- | :---------------------------------------- |
| **핵심 기술**   | Python 스크립트 + Playwright 직접 제어       | LLM Agent + MCP 서버를 통한 간접 제어     |
| **제어 주체**   | 작성된 코드 (Rule-based)                     | LLM (AI 모델)                             |
| **데이터 수집** | **Network Interception (GraphQL 패킷 캡처)** | (일반적으로) DOM 스크래핑 위주            |
| **비용**        | 무료 (로컬 실행)                             | 유료 (LLM 토큰 비용 발생)                 |
| **속도**        | 매우 빠름 (기계적 루프)                      | 느림 (매 행동마다 LLM 추론 필요)          |

---

## 2. 상세 비교 분석

### 🔍 핵심 차이점 1: 네트워크 인터셉트 (Network Interception)

가장 결정적인 차이점입니다. 현재 `sns_crawler`는 화면에 보이는 요소가 아니라 **네트워크 패킷(GraphQL Response)**을 가로채서 데이터를 얻습니다.

- **기존 방식**: `page.on("response", handler)`를 통해 모든 네트워크 응답을 실시간으로 감지하고, 특정 패턴의 JSON을 파싱하여 저장합니다. 이는 Playwright의 저수준(Low-level) API를 사용합니다.
- **MCP 방식**: `remorses/playwriter`는 "클릭", "입력", "스크롤"과 같은 고수준(High-level) 도구를 LLM에게 제공하는 데 초점이 맞춰져 있습니다. 현재 대부분의 브라우저 MCP는 **실시간 네트워크 패킷 캡처 및 대용량 JSON 파싱** 기능을 도구 형태로 완벽하게 제공하지 않습니다.
  - _가능성_: 설령 도구가 있다 하더라도, 수백 개의 게시물이 담긴 거대한 JSON 데이터를 LLM의 컨텍스트(Context Window)로 가져와서 처리하는 것은 비효율적이며 비용이 많이 듭니다.

### ⚙️ 핵심 차이점 2: 제어 흐름 (Control Flow)

- **기존 방식 (Loop)**:

  ```python
  while not stop_condition:
      scroll_down()
      process_packets()
      if found_stop_code: break
  ```

  매우 빠르고 안정적입니다.

- **MCP 방식 (Agentic Loop)**:
  1. Agent: "스크롤을 좀 내려봐" (LLMCall)
  2. Tool: (스크롤 실행) -> 결과 리턴
  3. Agent: "새로운 데이터가 로딩됐는지 확인해" (LLMCall)
  4. Tool: (화면 분석) -> 결과 리턴

  게시물 100개를 수집하기 위해 수십 번의 LLM 왕복(Round-trip)이 필요하며, 이는 **속도 저하**와 **토큰 비용 폭탄**으로 이어질 수 있습니다.

### 🛠️ 유지보수 및 유연성

- **기존 방식**: 사이트 구조(HTML Class 등)가 바뀌면 코드를 수정해야 합니다. 하지만 현재 방식은 HTML이 아닌 API 응답을 보므로 UI 변경에 비교적 강합니다.
- **MCP 방식**: UI 변경에 매우 강합니다. "저장된 버튼을 눌러"라고 하면 selector가 바뀌어도 시각적/의미적으로 찾아낼 수 있습니다. 하지만 대량 수집 업무보다는 **복잡한 단일 상호작용**(예: 쇼핑몰 결제, 회원가입 등)에 더 적합합니다.

## 3. 결론 및 제언

### ❌ `sns_crawler` 대체용으로는 비추천

**대량의 데이터를 반복적으로 수집**하는 현재의 목적(`sns_crawler`)에는 `remorses/playwriter` MCP 방식이 적합하지 않습니다.

1. **속도**: 훨씬 느립니다.
2. **비용**: 불필요한 LLM 토큰을 과도하게 소모합니다.
3. **기능 제한**: 핵심 기능인 'GraphQL 네트워크 캡처'를 MCP 도구만으로 구현하기가 매우 까다롭거나 불가능할 수 있습니다.

### ✅ 활용 가능한 시나리오

만약 `sns_crawler`의 목적이 데이터 수집이 아니라, **"특정 게시물에 댓글을 달거나, 설정을 변경하는 등 복잡한 UI 조작"**이라면 `remorses/playwriter`가 훌륭한 대안이 될 수 있습니다.

> **💡 요약 Advice**
>
> 현재의 빠르고 강력한 데이터 수집 파이프라인(`sns_crawler`)은 기존의 **Pure Python + Playwright** 방식을 유지하는 것이 기술적으로나 비용적으로 훨씬 우수합니다. MCP는 코딩을 도와주는 도구로 활용하거나, 단발성 에이전트 작업에 사용하는 것을 권장합니다.

## 4. 실증 테스트 결과 (2026-01-30 업데이트)

최근 수행한 `scrape_with_playwriter.js` (Playwright Persistent Context 방식) 테스트 결과, 초기 우려와 달리 특정 측면에서 매우 유의미한 장점이 확인되었습니다.

### ✅ 검증된 장점

1. **데이터 품질의 완벽한 일치**
   - 기존 Python `sns_crawler`와 새로운 Node.js Playwright 스크립트의 결과물을 정밀 비교(5개 샘플)한 결과, 텍스트 내용이 **100% 일치**하거나 타임스탬프 표기 차이 정도의 미세한 차이만 발생했습니다.
   - 이는 스크립트 방식 변경으로 인한 **데이터 품질 저하가 없음**을 증명합니다.

2. **탁월한 인증 안정성 (Bot Detection 회피)**
   - 기존 방식: 세션 쿠키를 json 파일로 저장/로드하는 방식. 종종 세션 만료나 재로그인 이슈 발생.
   - **New 방식**: `user-data-dir`을 지정하여 평소 사용하는 브라우저 프로필을 그대로 로드.
   - **효과**: 별도의 로그인 과정 없이도 인증이 유지되며, 실제 사용자 환경과 동일하게 인식되어 **섀도우밴(Shadowban)이나 로그인 챌린지**를 피할 확률이 획기적으로 낮아집니다.

### ⚠️ 기술적 과제 (To-Do)

1. **브라우저 컨텍스트 연결 (Attach vs Launch)**
   - 실행 중인 브라우저에 연결하거나 새 페이지를 찾을 때 `No suitable page found` 오류가 발생할 수 있습니다.
   - **해결책**: 견고한 예외 처리 및 페이지 탐색 로직(Target Discovery) 고도화 필요.

2. **대량 데이터 처리 검증**
   - 소량 데이터(5개) 테스트는 성공했으나, **Infinite Scroll** 상황에서의 DOM 가상화 대응 및 메모리 누수 여부는 추가 검증이 필요합니다.

### 💡 수정 제언 (Revised Recommendation)

기존 결론("비추천")을 수정합니다.
대량 수집의 속도는 Python 패킷 캡처 방식이 여전히 우위일 수 있으나, **안정적인 봇 탐지 우회와 장기적인 유지보수**를 위해서는 **Playwright Persistent Context 방식(Node.js/Python 무관)**으로의 전환을 강력히 고려해야 합니다.

특히, "인증이 자꾸 풀리는 문제"를 해결하고 싶다면 이 방식이 **Game Changer**가 될 수 있습니다.

## 5. 추가 레슨: Playwright (독립 실행) vs Playwriter (확장 프로그램 연결)

2026-01-30 분석을 통해 확인된 두 방식의 결정적 차이와 선택 가이드입니다.

### 📊 방식별 상세 비교

| 구분            | **Playwright 독립 실행** (`Launch`)           | **Playwriter 확장 프로그램 연결** (`Attach`)    |
| :-------------- | :-------------------------------------------- | :---------------------------------------------- |
| **제어 방식**   | `chromium.launch()` (새 프로세스 실행)        | `connectOverCDP()` (실행 중인 브라우저 연결)    |
| **자동화 수준** | **최상**: 스크립트 시작 시 브라우저 자동 기동 | **중**: 사용자가 브라우저를 켠 후 스크립트 실행 |
| **로그인/인증** | **중**: `auth.json` (쿠키) 수동 관리 필요     | **최상**: 이미 로그인된 브라우저 세션 100% 활용 |
| **환경 구성**   | **하**: 전용 브라우저 바이너리 수동 설치 필수 | **상**: 기존 브라우저 + 확장 프로그램만 필요    |
| **실행 엔진**   | 스크립트 전용 독립 브라우저                   | 사용자의 실제 메인 브라우저                     |

### 💡 핵심 교훈 (Lesson Learned)

1.  **브라우저 실행 권한의 한계**:
    - `Playwriter`(확장 프로그램 방식)는 이미 동작 중인 브라우저에 "숟가락을 얹는" 구조이므로, 물리적으로 브라우저(`.exe`)를 새로 실행시키는 기능은 없습니다.
    - 반면, `Playwright` 독립 방식은 브라우저를 직접 켤 수 있지만, 실행을 위해 전용 바이너리 설치가 선행되어야 하며 로그인이 풀린 상태로 시작됩니다.

2.  **적정 도구 선택 기준**:
    - **완전 무인 자동화**가 목표라면: `Playwright 독립 실행` + `auth.json` 로드 방식이 유리합니다.
    - **강력한 봇 탐지 우회**와 **편리한 로그인 유지**가 목표라면: `Playwriter 연결` 방식이 압도적으로 유리합니다. (사용자가 브라우저만 켜두면, 이후 페이지 이동/스크롤/수집은 완전 자동 가능)

3.  **최종 전략**:
    현재 Threads 수집과 같이 인증 장벽이 높은 상황에서는 **Playwriter 연결 방식**을 유지하되, 스크립트 시작 시 `page.goto('https://www.threads.net/saved')`를 통해 **페이지 이동 자동화**까지만 구현하는 것이 가장 효율적인 균형점입니다.

## 6. 전략적 판단: 기존 Python 방식(패킷 캡처) vs 새로운 JS 방식(DOM 스크래핑)

기존 자산인 `threads_scrap.py`와 새로운 하이브리드 방식의 실익을 최종 비교 분석합니다.

### 📊 기술적 우위 분석

| 비교 항목         | **기존 방식 (`threads_scrap.py`)**      | **새로운 JS/Playwriter 방식**        |
| :---------------- | :-------------------------------------- | :----------------------------------- |
| **추출 기술**     | **네트워크 패킷 캡처 (GraphQL)**        | **DOM 스크래핑 (HTML 텍스트)**       |
| **데이터 정밀도** | **최상** (원본 이미지, 고유 ID 등 포함) | **중** (노이즈 많음, '더 보기' 필요) |
| **안정성**        | UI가 바뀌어도 API만 유지되면 정상 작동  | UI(클래스명 등) 변경 시 즉시 실패    |
| **로그인 관리**   | `auth.json` 직접 수집 (번거로움)        | 실제 브라우저 세션 활용 (매우 편리)  |

### 💡 하이브리드 제안(`Playwright` + `Playwriter`)에 대한 평가

- **실익 부족**: `Playwright`로 브라우저를 직접 켤 수 있는데 그 위에 `Playwriter`를 얹는 것은 복잡도만 높이고 속도를 늦추는 행위 (대리운전 기사 소환과 유사).
- **기능 중복**: 이미 제어 권한을 얻은 상태에서 중간 매개체를 두는 것은 비효율적임.

### 🚀 최종 제언 (Sweet Spot 전략)

기존 `threads_scrap.py`의 강력한 성능과 `Playwriter`의 로그인 편의성을 결합한 **최적의 개선 방향**은 다음과 같습니다.

1.  **추출 로직 유지**: 강력한 **패킷 캡처(Network Interception)** 로직은 유지합니다.
2.  **로그인 방식 개선**: `auth.json` 파일을 로드하는 대신, **실제 브라우저 프로필 경로(`user-data-dir`)를 직접 참조**하여 브라우저를 기동합니다. (단, 환경에 따라 브라우저 프로세스 잠금 등의 불안정성이 발생할 수 있음이 확인됨)

## 7. 최후의 승부수: Playwriter(JS) + 패킷 캡처(Network Interception)

실제 환경 테스트 결과, `Persistent Context`(Python v2) 방식이 브라우저 프로세스 충돌 및 제어권 상실 이슈를 일으킬 경우에 대비한 **가장 현실적이고 강력한 최종 대안**입니다.

### 🛠️ 구조적 혁신: "안정성" + "정밀도"의 결합

| 핵심 요소       | **선택된 기술**                      | **역할/이점**                                                    |
| :-------------- | :----------------------------------- | :--------------------------------------------------------------- |
| **연결 도구**   | `Playwriter` (JS Extension)          | 브라우저 충돌 없이 켜져 있는 탭에 즉시 연결 (프로세스 잠금 우회) |
| **추출 기술**   | `Packet Capture` (Response Listener) | DOM 분석 대신 백그라운드 JSON 데이터를 직접 탈취 (고정밀도 확보) |
| **로그인 관리** | 사용자 브라우저 세션                 | `auth.json` 관리나 자동 로그인 실패에 대한 리스크 제로           |

### 🚀 최종 권고

- `threads_scrap_v2.py` (Python) 방식이 환경적 요인으로 작동하지 않을 경우, **`scrape_with_playwriter.js`에 네트워크 핸들러 로직을 이식**하는 것이 최선의 선택입니다.
- 이 방식은 **"브라우저 충돌 이슈 완전 봉쇄" + "로그인 과정 생략" + "고정밀 원본 데이터 수집"**을 동시에 만족하는 유일한 경로입니다.

## 9. 추가 실험: '재시작 없는 완전 자동화(Sweet Spot)' 시도 및 실패 기록 (2026-01-30)

사용자의 Chrome 브라우저를 재시작하지 않고 Threads 데이터를 수집하는 최적의 지점을 찾기 위한 심화 실험 결과입니다.

### 🧪 실험 가설 및 검증

1.  **가설 1: Playwright 라이브러리 제약**
    - `launchPersistentContext` 사용 시 기존 프로필 충돌 및 포트 점유 이슈 발생.
    - **결과**: ✅ **사실**. 라이브러리 직접 기동은 무한 타임아웃 발생으로 사용 불가.
2.  **가설 2: OS 레벨 Spawn 우회**
    - OS 명령어로 Chrome을 직접 실행(9222 포트 개방)하여 연결 시도.
    - **결과**: ⚠️ **부분 성공**. 브라우저는 실행되나 자동화 제어 연결이 매우 불안정함.
3.  **가설 3: 세션 유지**
    - 디버깅 모드 실행 시 기존 로그인 세션을 그대로 사용할 수 있을 것임.
    - **결과**: ❌ **실패**. 동일 경로 지정에도 불구하고 별도 프로필이 생성되어 재로그인 요구됨.

### 🔬 시도한 방법 및 결과

- **방법 1: launchPersistentContext (표본 방식)**: ❌ 15초 타임아웃 및 프리징.
- **방법 2: OS Spawn + connectOverCDP**: ⚠️ 연결은 되나 **데이터 수집 0개**(네트워크 패킷 캡처 작동 안 함).
- **방법 3: PowerShell 직접 실행**: ⚠️ 연결 성공했으나 세션이 초기화되어 다시 로그인해야 함.

### 🚧 발견된 핵심 기술 장벽

1.  **프로필 독점 제어**: Chrome은 보안상 동일 프로필을 두 프로세스가 공유하는 것을 원천 차단함.
2.  **CDP 포트 개방 타이밍**: 브라우저 실행 후 포트가 준비되는 시점이 매번 달라 연결 안정성이 매우 낮음.
3.  **네트워크 리스너 상실**: 직접 연결 방식에서는 `page.on('response')` 이벤트가 간헐적으로 유실되는 치명적 버그 확인.

## 10. 최종 결론 및 권고 사항 (최종본)

**"재시작 없이 안정적으로 작동하는 100% 자동화는 현재 기술적으로 득보다 실이 많음."**

1.  **메인 솔루션**: **v1 (Playwriter 확장 프로그램 연결)** 방식을 유지합니다.
    - **이유**: 이미 30개 수집에 성공했으며, 브라우저 세션을 완벽하게 재사용할 수 있는 유일하고 안전한 경로입니다.
    - **UX**: 사용자가 브라우저를 켜고 아이콘을 1회 클릭하는 수고는 있지만, **데이터 정밀도와 보안 우회** 측면에서 압도적입니다.

2.  **실험 종료**: '재시작 없는 완전 자동화(Spawn 방식)'에 대한 추가 시도는 실익이 없으므로 중단합니다.

3.  **향후 방향**: 만약 더 높은 자동화가 필요하다면, '브라우저 제어'보다는 '인증 세션(auth.json) 관리 자동화'에 집중하는 것이 더 현실적인 접근입니다.

> **🏁 최종 요약**
>
> 억지로 브라우저를 억지로 통제하려 하기보다, **사용자의 브라우저를 그대로 활용하는 v1(Attach 방식)**이 현재 Threads 수집 환경에서는 가장 현명하고 안정적인 "스윗스팟"입니다.
