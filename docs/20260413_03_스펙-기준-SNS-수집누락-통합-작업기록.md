# SNS 수집누락 스펙 기준 통합 작업 기록

## 발단: 사용자 요청

웹 뷰어 `index.html`의 업데이트 버튼을 눌렀는데 실제 SNS 업데이트가 안 된 것처럼 보인다는 문제 제기에서 시작했다. 최초 관심사는 버튼 클릭이 실제 수집을 실행했는지, 그리고 그 실행 이력이 로그와 산출물에 남았는지 확인하는 것이었다.

이 문서는 Threads만 따로 떼어 적는 작업 기록이 아니다. `docs/specs/20260412_01_SNS-스크래퍼-수집누락-개선.md`에 포함된 전체 작업 범위를 기준으로, 플랫폼별 판단과 실제 수정·검증·커밋 결과를 한 번에 정리한다.

## 작업 상세내역

### 1. 초기 진단과 로그 확인

로그 3종과 `output_total/total_full_20260412.json` 생성 시각을 대조한 결과, 2026-04-12 16:32~16:34 KST 실행분이 서로 정합했다. 즉 웹 뷰어의 업데이트 버튼 클릭은 실제 스크래핑 실행으로 이어졌고, 실행 흔적도 남고 있었다.

문제는 로그 파일이 `"w"` 모드로 열려 이전 실행 이력이 매번 사라진다는 점이었다. 이 항목은 추적성 결함으로 판단되어 스펙의 R1 범위에 포함됐고, 현재 `total_scrap.py`에서 `open(log_path, "a")`로 반영된 상태다.

### 2. 플랫폼별 진단 결과

스펙 문서와 기존 세션 정리를 교차 확인한 결과, 플랫폼별 결론은 아래와 같이 정리된다.

| 플랫폼/항목 | 상태 | 근거 |
|-------------|------|------|
| Threads | ❌ 실제 누락 | 지정한 누락 게시물이 플랫폼 파일, update 결과, 실패 목록 어디에도 없었고, update 중단 로직이 구조적으로 조기 종료를 만들고 있었다 |
| LinkedIn | ✅ 이슈 없음 | 플랫폼 파일, `total_full`, `data.js`, 뷰어 기본 정렬 기준에서 모두 정상 표시로 확인됐다 |
| Twitter | ⚠️ 오염 확인 | `platform_id`는 존재하지만 본문이 다른 게시물 텍스트로 오염됐고, 동일 본문 경고가 다수 발생했다 |

<span style="color:#888">*정렬 기준: 실제 수정 우선순위와 사용자 체감 영향 순서다. Threads는 직접 수정 대상, Twitter는 후속 과제, LinkedIn은 닫힌 이슈다.*</span>

### 3. 스펙 범위에서 실제로 완료된 항목

스펙 문서 기준으로 이번 작업 단위에서 실제 반영이 끝난 항목은 세 가지다.

- `R1` 로그 보존: `total_scrap.py`에서 log append 모드 반영
- `R3` Twitter 본문 오염: `BACKLOG.md` 후속 과제로 이관
- `R4` LinkedIn: 스크래퍼/뷰어 정상으로 결론 내리고 코드 변경 없이 종료

`R2` Threads는 이 범위의 핵심이지만, 스펙 기준으로는 아직 “완료”가 아니라 “부분완료”다. 이유는 후속 구현으로 신규 수집은 복구됐지만, 스펙이 성공 기준에 넣은 특정 누락 code 검증이 아직 남아 있기 때문이다.

### 4. Threads 후속 구현과 검증

기존 세션에서는 `stop_codes` 즉시 중단 로직이 근본 원인이라고 파악했고, 실측 저장 목록에서 신규 5건 뒤에 기수집 16건 연속 구간이 확인되어 `기수집 연속 카운터(N=20)` 접근이 채택됐다. 이후 후속 구현에서 `thread_scrap.py`에 아래 보강이 실제 반영됐다.

- `existing_codes` 기반 기수집 skip
- `consecutive_dup_count`, `CONSECUTIVE_DUP_LIMIT` 추가
- `scroll_had_response` 도입으로 “응답은 왔지만 모두 기수집 skip”일 때 `no_new_data_count`가 올라가지 않게 보정
- 네트워크/DOM 양쪽에서 연속 기수집 한도 도달 시 중단

실행 검증에서는 `update only` 모드에서 조기 종료 패턴이 사라졌고, `output_threads/python/update/threads_py_simple_update_20260413_090814.json`와 `output_threads/python/update/threads_py_simple_update_20260413_091120.json`이 생성됐다. 특히 두 번째 실행에서는 신규 15건이 수집되어 최종 `threads_py_simple_20260413.json`이 669건으로 갱신됐다.

다만 스펙 성공 기준에 포함된 특정 code `DQ8mqRaEtUL`, `DWn9J1mEpb4`는 현재 결과 파일에서 확인되지 않았다. 따라서 Threads는 “수집 복구 신호 확인”까지는 완료됐지만, “스펙이 요구한 기대 항목 충족”까지는 아직 닫히지 않았다.

### 5. 사후 판정과 저장소 반영

후속 세션에서는 수정 전에 제안됐던 두 가지 크리티컬 이슈에 대해서도 사후 판정이 이뤄졌다. 결론은 둘 다 실제 문제였고, 다른 세션에서 수정된 것이 맞다는 방향으로 정리됐다.

- `consecutive_dup_count` / `CONSECUTIVE_DUP_LIMIT` 초기화 누락
- `scroll_had_response` 초기화 위치 문제

최종 저장소 반영은 아래 두 커밋으로 나뉘어 처리됐다.

- `80b8609` `fix(threads): update 모드 기수집 연속 스킵 로직 보강`
- `cf517fd` `chore(docs,gitignore): 세션 진행 문서 추가와 playwright-mcp 제외`

두 커밋 모두 `main`에 push 완료됐고, `.playwright-mcp/`는 런타임 산출물로 판단해 `.gitignore`에 추가했다.

## 리스크 및 미해결 이슈

- Threads는 신규 수집이 일부 복구됐지만, 스펙 성공 기준에 포함된 특정 code 검증이 아직 남아 있다.
- `all` 모드는 제한된 smoke run만 수행했으므로, 전체 50회 스크롤 기준의 완전 회귀 검증은 아니다.
- Twitter 본문 오염은 원인 파악까지만 끝났고, 수집 재시도 또는 오염 감지 로직은 미구현 상태다.
- 현재 세션에는 자동 연결된 plan 파일이 없으므로, Threads 후속 판단은 스펙 문서와 세션 정리 문서를 함께 봐야 맥락이 온전히 복원된다.

## 다음 액션

- Threads 기대 code가 실제로 어디에서 누락되는지 추가 추적한다. 우선순위는 네트워크 응답 포함 여부, 필터 탈락 지점, `existing_codes` 오인 여부 순서다.
- `all` 모드의 전체 회귀 검증이 필요하면 제한 없는 런타임으로 한 번 더 확인한다.
- Twitter 본문 오염은 `BACKLOG.md` 항목을 기준으로 별도 plan을 세워 재시도/오염 감지 전략을 분리 설계한다.
- 이후 `doc-act`로 이어갈 경우, 이번 문서를 기준 작업기록으로 두고 검증 결과만 추가 갱신한다.
