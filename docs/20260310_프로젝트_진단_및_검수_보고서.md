## 1. 프로젝트 진단 요약

현재 프로젝트는 **"스크래퍼 자체의 기능"**보다 **"운영 계약 및 검증 체계"**에서 크리티컬한 리스크가 발견되었습니다. 주요 불일치 사항은 다음과 같습니다.

- 서버 API 실패 경로와 실제 로그 처리의 불일치
- 통합 메타데이터 계약(JSON Key)과 서버 로직의 불일치
- 설치 문서(README)와 실제 저장소 파일 구조의 불일치
- 테스트 자산이 단순 출력형 스크립트에 머물러 있어 자동 회귀 검증이 불가능함

---

## 2. 검증 결과 (Verification Results)

진단 내용을 바탕으로 실시한 사전 검증 결과는 다음과 같습니다.

- **컴파일 테스트**: `python -m py_compile` 명령을 통한 구문 오류 검사 통과
- **설정 검증**: `.agent/scripts/validate_agent_config.py` 실행 결과 정상
- **서버 API 테스트**:
    - `/api/status`, `/api/latest-data`: 정상 응답 확인
    - `/api/run-scrap`: 잘못된 `mode` 전달 시 500 에러 발생 (NameError로 인한 비정상 종료)
- **기존 테스트 검토**: `test_logic_refinement.py` 등 기존 스크립트들이 `assertion` 없는 출력형 스크립트로 확인되어 자동 회귀 테스트로서의 가치가 낮음

## 3. 크리티컬 개선 포인트 (P0)

검수 과정에서 즉시 수정이 필요한 3가지 핵심 이슈를 확인했습니다.

### P0-1: server.py 실패 분기 stderr_val 참조 버그
- **현상**: 스크래핑 실패 시 원래의 에러 로그 대신 `NameError: name 'stderr_val' is not defined`가 발생하여 디버깅이 완전히 차단됨.
- **원인**: `server.py:107`에서 `stderr=subprocess.STDOUT`으로 설정하여 stderr가 stdout에 병합되었으나, `151행`과 `156행`에서 존재하지 않는 `stderr_val` 변수를 참조함.
- **조치**: `stderr_val`을 `stdout_val`로 교체하거나, 실패 시에도 전체 출력을 포함하도록 수정 필요.

### P0-2: 메타데이터 계약 불일치
- **현상**: 서버 대시보드 및 API에서 통계 데이터가 항상 0으로 표시됨.
- **원인**: 
    - 서버(`server.py:137-139`)가 읽는 키: `new_items_count`, `new_threads_count`, `new_linkedin_count`
    - 스크래퍼(`total_scrap.py:281-284`)가 쓰는 키: `total_count`, `threads_count`, `linkedin_count`, `twitter_count`
- **조치**: `server.py`가 실제 산출물 메타데이터 키를 읽도록 동기화 필요.

### P0-3: import 시점 argparse 실행 구조
- **현상**: 모듈을 import하는 것만으로 `parse_args()`가 실행되어, 테스트 러너나 외부 호출 시 `SystemExit`와 함께 프로세스가 종료됨.
- **원인**: `thread_scrap.py:49-51`, `linkedin_scrap.py:22-24`, `total_scrap.py:61` 등에서 전역 범위(module level)에 파서 실행 코드가 위치함.
- **조치**: 모든 실행 로직을 `main()` 함수로 캡슐화하고 `if __name__ == "__main__":` 블록을 사용하도록 리팩터링 필요.

## 3. 구현 검증 테스트 계획

향후 안정적인 운영을 위해 다음과 같은 테스트 체계를 도입할 예정입니다.

1. **계약 테스트 (Contract Testing)**
    - `server.py` API 응답 스키마 고정 및 검증
    - `/api/status`, `/api/latest-data`, `/api/run-scrap`의 성공/실패 케이스별 상태 코드 및 JSON 구조 확인

2. **순수 로직 단위 테스트 (Unit Testing)**
    - **Threads**: `clean_text`, `parse_relative_time`, 중복 제거(dedupe) 로직
    - **LinkedIn**: `extract_urn_id`, 이미지 추출 재귀 로직
    - **Total**: 포스트 재정렬, 병합(merge) 및 정렬 키 로직
    - *입력은 Mock/Fixture JSON을 사용하여 브라우저 없이 실행 가능하도록 구성*

3. **파일 계약 테스트 (File Schema Validation)**
    - 각 플랫폼별 산출물 JSON(`threads_py_full_*`, `total_full_*` 등)의 필수 필드 존재 여부 검증
    - 서버가 기대하는 메타데이터와 스크래퍼가 생성하는 메타데이터의 일치 여부 상시 검증

4. **스모크 테스트 (Smoke Testing)**
    - Playwright 기반의 짧은 런(`--mode update`, `--limit 1`)을 통해 브라우저 연동 및 인증 상태 확인
    - 결과물 파일 생성 및 필수 필드 채움 여부만 확인하여 실행 시간 단축

## 4. 우선순위 및 향후 조치

1. **P0 (최우선)**: `server.py` 실패 분기 및 메타데이터 키 수정, `argparse` 구조 개선
2. **P1 (중요)**: README 설치/실행 문서 최신화, assertion 기반 자동 테스트 초기 세트 도입
3. **P2 (보통)**: 브라우저 기반 smoke 테스트를 선택적 실행 환경(CI 등)으로 분리

## 5. 가정 및 기본값

- **실제 수집 검증의 한계**: 인증이 필요한 SNS 수집은 완전 자동화가 어렵기 때문에, 브라우저 기반 검증은 smoke 레벨로 제한하여 운영합니다.
- **회귀 Fixture**: 기존 산출물 JSON 파일을 회귀 테스트의 fixture로 재사용하는 것을 기본으로 합니다.
- **테스트 프레임워크**: Python 표준 `unittest`보다 `pytest`가 적합하지만, 당장의 목표는 프레임워크 선택보다 "import-safe 구조"와 "assertion이 포함된 자동 테스트"를 확보하는 데 집중합니다.
