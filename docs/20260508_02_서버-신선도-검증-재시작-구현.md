# SNS 뷰어 서버 신선도 검증 재시작 구현

## 발단: 사용자 요청

태그 관리 통합 배포 직후 브라우저 콘솔에서 `script.js?v=20260508-tag-catalog:1858 Data load failed: SyntaxError ... "<!doctype "... is not valid JSON` 오류 발생. 확인 결과 새 프론트엔드는 `/api/get-tag-catalog`를 호출하지만, 실행 중인 Flask 서버는 구버전 프로세스라 해당 API 대신 루트 `index.html`을 반환함.

사용자 질문: "원래는 실행하면 기존 프로세스 죽이고 다시 시작하도록 안되어 있나?" 확인 결과 `sns_hub.vbs`는 `/api/status`가 200이면 기존 서버를 그대로 사용하고, `run_viewer.bat`도 기존 5000번 포트를 먼저 종료하지 않음. 이에 사용자가 "계획 수립해서 보고해줘" 요청 후 `doc-act` 실행 요청.

## 작업 상세내역

확인된 현재 동작:

- `sns_hub.vbs`: `http://localhost:5000/api/status`가 200이면 서버가 준비됐다고 판단하고 브라우저만 실행.
- `run_viewer.bat`: `start /b python server.py`를 호출하지만 기존 5000번 포트 점유 프로세스 종료 없음.
- `stop_viewer.bat`: 별도 수동 중지 스크립트이며, 실행 시작 흐름에 자동 연결되지 않음.
- `server.py`: 미등록 `/api/*` 경로가 catch-all 라우트에 걸리면 SPA fallback으로 `index.html`을 반환할 수 있음.

실행 계획:

1. `server.py`에서 미등록 `/api/*` 경로는 JSON 404를 반환하도록 변경.
2. `scripts/restart_viewer_server.ps1` 신규 작성. 필수 API(`/api/status`, `/api/get-tag-catalog`) 신선도 확인 후, 실패 시 5000번 포트의 `server.py` 프로세스만 종료하고 재시작.
3. `sns_hub.vbs`가 PowerShell helper를 호출하도록 변경. helper가 없을 때만 기존 방식으로 fallback.
4. `run_viewer.bat`도 같은 helper를 호출하고 루트 URL을 열도록 정렬.
5. 정적/통합 테스트와 PowerShell parser 검증으로 확인. 실제 실행 중 프로세스 kill 검증은 사용자 명시 승인 없이는 수행하지 않음.

## 의사결정 기록

| 옵션 | 판단 | 근거 |
|------|------|------|
| 무조건 5000번 프로세스 종료 후 재시작 | 기각 | 실행마다 서버가 끊기며, 수집 중 작업을 불필요하게 중단할 수 있음 |
| `/api/status`만 확인 | 기각 | 이번 장애처럼 구버전 서버도 200을 반환할 수 있음 |
| 필수 API 신선도 확인 후 선택적 재시작 | 채택 | 새 프론트와 구버전 서버 조합을 막으면서 정상 서버는 유지 가능 |
| 모든 `python.exe` 종료 | 기각 | TickTick MCP, Notion MCP 등 무관한 Python 프로세스를 종료할 위험 |

- 결정: `필수 API 확인 → 구버전이면 5000번 server.py만 재시작`.
- 근거: 장애 원인이 "서버 생존 여부"가 아니라 "서버 코드 신선도"였기 때문.
- 트레이드오프: 런처가 약간 복잡해지지만, 재시작 범위를 좁혀 무관 프로세스 종료 위험을 줄임.

## 리스크 및 미해결 이슈

- 실제 서버 프로세스 kill 검증은 사용자 명시 승인 없이는 생략 필요.
- 이미 떠 있는 구버전 서버는 코드 변경만으로 자동 교체되지 않음. 변경 후 `sns_hub.vbs`를 다시 실행해야 helper가 동작함.
