# SNS 조회 CLI 구축 — 설계·검수·구현·검증 전과정

## 발단: 사용자 요청

현재 프로젝트의 SNS 수집 내역을 Claude Code 세션에서 수시로 조회할 수 있는 `api`, `cli`, `skill` 계층을 만들고 싶다는 요청에서 시작했다. 사용자는 동기화, 태그 CRUD, 키워드 조회, 조회 결과 기반 분석 리포트 저장까지 한 흐름으로 다루고 싶어 했고, 데이터 규모가 약 1,000건일 때 SQLite 전환이 필요한지도 함께 검토 대상이었다.

초기 결론은 "지금은 SQLite 전환 없이 JSON 기반 조회가 타당하다"였다. 다만 이 판단은 구현 단순화만으로 끝나지 않았고, 이후 Codex 검수에서 태그 write 경로와 분석 샘플링 설계가 실제 리스크로 드러나면서 Phase 분리 설계로 이어졌다.

## 작업 상세내역

### 1. 초기 설계와 plan 작성

초기 설계는 `query-sns.mjs` CLI와 `sns-query` 스킬을 중심으로 잡았다. 조회 요구를 `recent`, `get`, `search`, `by-platform`, `by-user`, `stats`, `by-tag`, `tag list` 같은 서브커맨드로 매핑하고, 자연어 요청은 스킬이 CLI 호출로 변환하는 구조로 정리했다.

이 단계에서 SQLite 전환은 보류했다. 근거는 `982건 / 2.8MB` 규모에서는 JSON 메모리 로드가 충분히 빠르고, 실제 세션 검토에서도 수십 ms 수준으로 판단되었기 때문이다.

### 2. Codex plan 검수와 크리티컬 3건 수용

초기 plan은 Codex 검수에서 "목적 달성 가능성이 낮다"는 판정을 받았다. 핵심은 조회 전용 기능은 빠르게 구현 가능하지만, 태그 CRUD와 분석 리포트까지 한 번에 약속하면 정확성과 일관성이 무너진다는 점이었다.

수용한 크리티컬 이슈는 아래 3건이다.

| # | 이슈 | 수용 내용 |
|---|---|---|
| 1 | 태그 key canonicalization 부재 | `tag add/remove`를 Phase B로 이동 |
| 2 | 브라우저와 CLI의 last-writer-wins 충돌 | write path 설계를 Phase B 선행조건으로 분리 |
| 3 | 분석 리포트 `--limit 100` 고정 편향 | 고정 제거, `total_matches` 기반 판단으로 전환 |

검수보고서 문서는 `[[20260412_atomic-soaring-thacker_검수보고서]]`로 추적 가능하게 plan에 연결했다. plan 본문에도 검수 이력, 수용 이슈, 대응 방안을 표 형태로 반영했다.

### 3. Phase A/B 분리와 재계획

재작성된 plan은 범위를 명확히 분리했다.

- Phase A: 조회 전용 CLI + `tag list` 읽기 전용 + `sns-query` 스킬 + 자동화 검증
- Phase B: 태그 write 경로 전용 후속 작업

Phase B는 구현 전에 아래 4가지를 선행조건으로 명시했다.

1. canonical key 규칙 문서화
2. URL 정규화 로직 확정
3. write conflict 방안 확정
4. write-path 테스트 추가

이번 턴 범위에서는 Phase B를 구현하지 않고, BACKLOG 추적 항목으로 남겼다.

## Phase A: 조회 전용 CLI + 스킬 (이번 구현 범위)

### Step A1: 스캐폴딩 + 데이터 로딩

파일: `D:\vibe-coding\scrap_sns\utils\query-sns.mjs`

구조 (`query-sessions.js` 패턴 차용):
```
import (node:fs, node:path, node:url)
상수 (PROJECT_ROOT = import.meta.url 기반 ../, OUTPUT_TOTAL_DIR, TAGS_PATH)
유틸 (resolveLatestJson, loadPosts, loadTags, parseArgs, printUsage)
공통 (applyFilters, postToResult, formatBrief)
서브커맨드 함수 (cmd별 독립)
main() → parseArgs → switch → stdout JSON
```

핵심 함수:
- `resolveLatestJson()`: `fs.readdirSync` + `/^total_full_\d{8}\.json$/` 필터 + sort + 마지막
- `loadPosts()`: `readFileSync` + BOM 제거 (`replace(/^\uFEFF/, '')`) + `JSON.parse` → `{ data, file }`
- `loadTags()`: 파일 없으면 `{}` 반환. BOM 방어적 제거.
- `parseArgs()`: 수동 while 루프, 첫 non-option = command
- 에러는 stderr, 정상 결과는 JSON stdout

### Step A2: 읽기 전용 커맨드

| 커맨드 | 로직 |
|--------|------|
| `recent [N]` | applyFilters → created_at 내림차순 → limit (기본 10) |
| `get <platform_id>` | platform_id 정확매칭 → 없으면 code 필드 검색 → 전체 필드 + tags 반환 |
| `by-platform <name>` | twitter/트위터 → "x" 정규화, sns_platform 필터 |
| `by-user <username>` | username/display_name 대소문자 무시 includes |
| `stats` | 플랫폼별 건수, 날짜 범위, 태그 분포 top 20, 작성자 top 10, 파일 정보 |

공통 헬퍼:
- `applyFilters(posts, opts)`: --platform, --from, --to 적용
- `postToResult(post, tags)`: 요약 필드 추출. `media_count`는 `(post.media || []).length`로 계산.
- `get`만 전체 원본 + tags 반환

### Step A3: 검색 + 태그 필터 커맨드 (읽기 전용)

| 커맨드 | 로직 |
|--------|------|
| `search <keyword>` | 소문자 변환 → full_text, display_name, username, 태그에서 includes → match_fields 표시 |
| `by-tag <tag>` | tags 딕셔너리에서 해당 태그 가진 url 집합 → posts 필터 |
| `tag list [url]` | url 없음: 고유 태그 + 건수 / url 있음: 해당 url 태그 배열 (읽기만) |

### Step A4: brief 포맷 + 에지 케이스

- `--format brief`: JSON 대신 한줄 텍스트 (`[threads] @user (날짜) 텍스트 80자... [태그]`)
- 빈 결과, 파일 없음, 잘못된 인자 → stderr + exit(1)

공통 옵션: `--platform`, `--from`, `--to`, `--limit` (기본 10), `--format json|brief`

반환 JSON envelope:
```json
{ "command": "search", "keyword": "...", "total_matches": 47, "returned": 10, "posts": [...] }
```
