# SNS 태그 자동적용과 관리 통합

## 목적
신규로 추가된 SNS 게시물에 기존 태그가 자동 적용되지 않는 문제를 해결한다. 현재 설정 모달에는 `자동 태그`와 `태그 관리`가 분리되어 있지만, 실제 사용자는 둘을 하나의 `태그 관리` 경험으로 이해하기를 원한다.

트레이드오프: 태그를 하나의 개념으로 통합하면 사용성은 좋아지지만, 자동 매칭 범위를 넓히는 만큼 오탐과 대량 태그 변경 리스크를 별도 제어해야 한다.

## 요구사항
- R1. 신규·갱신 게시물은 기존 태그명과 본문/작성자 정보가 매칭될 때 자동으로 해당 태그를 받아야 한다.
- R2. `자동 태그`와 `태그 관리`는 하나의 `태그 관리` 탭으로 통합되어야 한다.
- R3. 기존 태그 데이터는 보존되어야 하며, `web_viewer/sns_tags.json`의 URL 기반 태그 매핑을 깨면 안 된다.
- R4. 현재 태그 CRUD와 태그 alias CRUD가 실제로 어떻게 구현되어 있는지 확인하고, 개선 필요 여부를 보고해야 한다.
- R5. 기존 `keyword -> tag` 자동 태그 규칙은 별도 규칙 개념이 아니라 각 태그의 alias/키워드 속성으로 통합 관리되어야 한다.

## 성공 기준
- `리서치` 태그가 이미 존재하는 상태에서, 본문에 `리서치`가 포함된 신규 LinkedIn 게시물에 `리서치` 태그가 자동 적용된다.
- 자동 적용 결과가 브라우저 `localStorage.sns_tags`와 `web_viewer/sns_tags.json`에 저장되어 새로고침 후에도 유지된다.
- 설정 UI에서 사용자가 한 곳에서 태그명, 자동 적용, alias/키워드, 강조 표시를 관리할 수 있다.
- 기존 per-post 태그 추가·삭제, 글로벌 태그 필터, primary 강조 표시가 회귀하지 않는다.

## 제약 조건
- 현재 분석 단계에서는 코드 수정을 하지 않는다.
- 태그 저장의 영구화 surface는 `web_viewer/sns_tags.json`이며, 브라우저 상태는 `localStorage.sns_tags`, `localStorage.sns_tag_types`, `localStorage.sns_auto_tag_rules`를 사용한다.
- 기존 태그 key는 canonical URL 기반이어야 한다. URL alias 정규화는 별도 기능이며 이번 SPEC 범위에서 다루지 않는다.
- 자동 태그 적용은 전체 게시물을 스캔하므로, 초기 로드마다 무제한 실행하면 성능과 의도치 않은 변경 문제가 생길 수 있다.

## 경계선
- [OK] 기존 태그명을 기본 자동 매칭 키워드로 사용하는 방향.
- [OK] 기존 `sns_tags.json` 구조를 즉시 갈아엎지 않고, 우선 현재 URL→태그 배열 구조를 유지한다.
- [OK] `자동 태그`와 `태그 관리` 탭은 하나의 `태그 관리` 탭으로 통합한다.
- [OK] 기존 `keyword -> tag` 자동 태그 규칙은 각 태그의 alias/키워드 속성으로 통합한다. 예: `심층리서치`, `research`도 `리서치` 태그의 alias/키워드로 관리.
- [OK] alias/키워드 저장 포맷은 별도 태그 카탈로그로 분리하고, 기존 `sns_auto_tag_rules`는 1회 변환한다.
- [OK] `autoEnabled` 플래그는 두지 않는다. 태그 카탈로그에 있는 태그는 기본적으로 자동 적용 대상이다.
- [OK] 단일 태그 row의 명시적 메뉴에서 태그명 변경과 태그 삭제를 제공한다.
- [NO] URL key 체계를 바꾸는 마이그레이션을 선행하지 않는다.
- [NO] URL alias 정규화 또는 Threads legacy URL key 마이그레이션은 이번 범위에 포함하지 않는다.
- [NO] 사용자 확인 없이 대량 태그 삭제, 대량 태그명 rename, alias 병합을 수행하지 않는다.

## 결정안
채택 방향은 "태그를 단일 관리 단위로 보고, 자동 적용은 태그 카탈로그의 기본 동작으로 둔다"는 것이다. 즉 사용자는 `리서치`라는 태그 하나를 관리하고, 그 태그에는 표시 속성(primary)과 매칭 키워드/alias가 붙는다.

UI는 별도 `자동 태그` 탭과 `태그 관리` 탭을 하나의 `태그 관리` 탭으로 합친다. 각 태그 row 또는 detail 영역에서 태그명, primary 강조, alias/키워드를 함께 관리한다.

단기 구현은 기존 구조를 최대한 보존한다. `sns_tags.json`은 그대로 URL→태그 배열로 두고, 기존 `keyword -> tag` 자동 태그 규칙은 1회 변환으로 태그 카탈로그의 alias/키워드에 편입한다. 자동 적용 시에는 태그명 자체와 alias/키워드를 함께 `/api/auto-tag/apply` 입력에 포함한다.

## 사실 확인
- `applyAutoTagRules()`는 `localStorage.sns_auto_tag_rules`에 저장된 수동 규칙만 서버에 보낸다. 위치: `web_viewer/script.js:1193`.
- `renderAutoTagRules()`는 기존 태그명으로 `Auto` 규칙을 화면에 만들어 보여주지만, 이 암묵 규칙은 실제 적용 API 입력에 포함되지 않는다. 위치: `web_viewer/script.js:2736`.
- 기존 `sns_auto_tag_rules`는 `keyword -> tag` 구조이므로, 기능적으로는 이미 태그 alias/키워드 역할을 할 수 있다. 다만 현재 UI에서는 별도 `자동 태그` 규칙으로 노출되어 태그 관리와 분리되어 보인다.
- per-post 태그 CRUD는 카드 내부에서 수행된다. 태그 추가는 입력/추천 클릭으로 `postTags[url]`에 push하고, 삭제는 해당 URL 배열에서 제거한다. 위치: `web_viewer/script.js:2290`.
- `태그 관리` 탭은 태그 생성·삭제·rename을 하지 않는다. 현재 기능은 태그 검색과 primary 강조 토글뿐이다. 위치: `web_viewer/script.js:2627`.
- 서버 태그 API는 `web_viewer/sns_tags.json` 전체를 읽고 쓰는 GET/POST만 제공한다. 위치: `server.py:594`, `server.py:607`.
- `/api/auto-tag/apply`는 전달받은 rules를 최신 posts에 매칭해 URL→태그 배열을 반환한다. 위치: `server.py:1119`.
- 현재 코드의 `aliasToCanonical`은 태그 alias가 아니라 Threads URL legacy key를 canonical URL로 옮기기 위한 URL alias다. 이번 SPEC의 태그 alias/키워드 범위와 별개다. 위치: `web_viewer/script.js:1255`.
- 현재 `web_viewer/sns_tags.json`은 1,021개 URL key와 17개 고유 태그를 가진다. 비 URL key는 발견되지 않았다.
