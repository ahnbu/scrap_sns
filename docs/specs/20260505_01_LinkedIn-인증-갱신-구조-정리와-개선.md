# LinkedIn 인증 갱신 구조 정리와 개선 SPEC

## 목적
LinkedIn 인증 갱신 실패와 관련해 `scrap_sns`, `sns_insight_update`, 공용 인증 런타임의 역할과 현재 장애 상태를 정리한다.

이 문서는 별도 세션의 AI가 이 문서만 보고도 다음 조사를 이어갈 수 있도록 만든 의사결정 앵커다. 핵심 목표는 "왜 현재 LinkedIn 수집이 막혔는가"를 확정하는 것이 아니라, 지금까지 확인된 사실과 남은 검증 항목을 분리해 다음 실험이 같은 혼선을 반복하지 않게 하는 것이다.

2026-05-05 KST 기준, 현재 `scrap_sns` LinkedIn 저장글 수집과 `sns_insight_update` LinkedIn 성과 수집은 같은 인증 파일을 사용하며, 해당 인증 파일은 LinkedIn에서 `checkpoint/challenge` 또는 `authwall`로 밀리고 있다. 단, 2026-05-05 11:10 KST에 일반 사용자가 여는 기본 Chrome에서는 saved-posts가 직접 열리는 것으로 확인됐다.

## 요구사항
- R1. 별도 세션이 문서만 보고도 LinkedIn 인증 갱신 문제의 배경, 진행 과정, 현재 상태를 이해할 수 있어야 한다.
- R2. `scrap_sns` LinkedIn 수집기, `scrap_sns` 인증갱신기, `sns_insight_update` LinkedIn 수집기의 역할 차이와 공통 인증 파일 의존성을 명확히 구분해야 한다.
- R3. 잘 동작하던 시기 대비 현재 인증 구조가 어떻게 바뀌었는지 비교해야 한다.
- R4. 현재까지 실행한 접근 방식별 테스트 결과를 사실과 가설로 분리해 기록해야 한다.
- R5. 사용자가 제공한 비교표와 과거/현재 인증구조 비교표를 부록으로 보존해야 한다.
- R6. 인증 갱신 안내문구는 실제로 유효한 갱신 방법을 찾은 뒤 수정해야 한다는 제약을 기록해야 한다.
- R7. 다음 세션의 실행 방향은 "기존 쿠키 재사용"이 아니라 "새 인증을 어떤 방식으로 만들면 유효한가" 검증으로 이어져야 한다.
- R8. SNS 인증 갱신과 수집의 성공·실패를 추적할 수 있는 구조화 로그가 필요하다.
- R9. LinkedIn, Threads, X 인증 갱신을 사용자가 더블클릭으로 시작할 수 있는 바로가기가 필요하다.
- R10. 인증 안내문구는 현재 인증 구조와 실행 경로에 맞도록 갱신되어야 하며, 이 작업은 TODO/백로그에서 추적되어야 한다.

## 성공 기준
- 이 문서에 현재까지 확인된 경로, 파일, URL, Playwright 방식, 실패 증상이 모두 포함되어 있다.
- 부록에 사용자가 제시한 3도구 비교표가 원래 의미를 잃지 않도록 Markdown 표로 보존되어 있다.
- 부록에 과거 구조와 현재 구조의 차이를 비교한 표가 포함되어 있다.
- 테스트 결과 표에 최소 다음 항목이 포함되어 있다: 기본 context, 인증갱신기 `--web` 유사 환경, `scrap_sns` 수집기 일치 환경, `sns_insight_update` 일치 환경, 실제 Chrome 채널, 임시 persistent profile + 기존 쿠키, 공식 `linkedin_scrap.py --mode update`.
- 다음 세션이 바로 실행할 수 있는 후속 검증 방향이 명확하다.
- 인증 성공·실패 로그 설계에는 최소한 플랫폼, 활동 유형, 인증 방식, 실행 경로, 결과, 실패 분류, KST 시각이 포함되어야 한다.
- 더블클릭 인증 바로가기 설계에는 `링크드인인증갱신-바로가기`, `쓰레드인증갱신-바로가기`, `트위터인증갱신-바로가기`가 포함되어야 한다.
- 인증 안내문구 갱신 작업이 후속 TODO에 명시되어 있어야 한다.

## 제약 조건
- LinkedIn은 보안·인증 정책이 외부에서 바뀔 수 있으므로, "원인 확정"은 로컬 코드만으로 단정하지 않는다.
- 현재 인증 파일 값이나 쿠키 값은 문서에 기록하지 않는다.
- `storage_state.json`을 무작정 덮어쓰지 않는다. 막힌 상태를 저장하면 장애 상태가 고정될 수 있다.
- 앱의 인증 버튼, `renew.py --web`, 대화형 PowerShell 갱신은 서로 다른 실행 경로로 취급한다.
- 인증 안내문구는 지금 바로 고치지 않는다. 먼저 실제로 유효한 갱신 절차를 확인한 뒤 반영한다.
- 인증 로그에는 쿠키값, 토큰값, challenge URL 전체, 개인정보성 식별자를 기록하지 않는다.
- 바로가기는 사용자가 직접 로그인·보안 인증을 수행할 수 있도록 visible PowerShell/브라우저 흐름을 띄우는 방식이어야 한다.
- 다음 구현 계획에서 인증·파싱 로직을 수정할 경우, 영구화 surface와 마이그레이션 여부를 별도로 명시해야 한다.

## 결정안
트레이드오프: 작은 수정은 `renew.py`의 LinkedIn 갱신 환경을 수집기와 맞추는 것이지만, 현재 테스트 결과는 기존 `storage_state.json` 자체가 이미 LinkedIn 서버 쪽 재검증 상태일 가능성을 보여준다.

채택된 접근은 다음 순서다.

1. 현재까지의 논의와 테스트 결과를 SPEC으로 고정한다.
2. 기존 `storage_state.json` 재사용, 새 `storage_state` 생성, Playwright 임시 persistent profile, 기존 `renew.py linkedin` interactive 방식은 실패로 고정하고 반복하지 않는다.
3. 일반 Chrome 기본 프로필에서는 saved-posts가 열리므로, 다음 실험은 일반 Chrome 프로필 기반 수집 구조 검토로 전환한다.
4. 어떤 방식이든 성공 기준은 saved-posts 직접 접근 가능 여부와 `linkedin_scrap.py --mode update` 또는 LinkedIn preflight 통과 여부다.
5. 성공 방식이 확인된 뒤에만 README와 앱 안내문구를 갱신한다.
6. 인증 문제 재발을 줄이기 위해 인증 갱신·수집 이벤트를 구조화 로그로 남기는 설계를 추가한다.
7. 사용자가 앱 버튼이나 AI 안내문구에 의존하지 않고 인증 갱신을 시작할 수 있도록 플랫폼별 더블클릭 바로가기를 설계한다.

### 현재 접근 환경

- `scrap_sns` LinkedIn 수집기는 `chromium.launch` + `new_context(storage_state)`를 사용한다.
- `scrap_sns` LinkedIn 수집기는 `Chrome/122` user agent, `ko-KR` locale, `900x500` viewport를 명시한다.
- `scrap_sns` 인증갱신기의 LinkedIn 경로는 `chromium.launch` + `new_context(storage_state)`이며, 현재 user agent와 locale을 명시하지 않는다.
- `sns_insight_update` LinkedIn 수집기는 `Chrome/122` user agent, `ko-KR` locale, `1280x1000` viewport를 명시한다.
- LinkedIn 쪽은 X와 달리 현재 persistent profile을 사용하지 않는다.

### 현재 장애 상태

- 2026-05-05 오전 기준, `sns_insight_update` LinkedIn preflight는 `saved_posts_ready=false`, `recent_authwall=true`, 카드/노출값 0건으로 실패했다.
- 2026-05-05 오전 기준, `scrap_sns` LinkedIn 저장글 수집도 실패 상태다.
- 공식 실행 `SNS_ORCHESTRATED_RUN=1`, `PYTHONIOENCODING=utf-8`, `python linkedin_scrap.py --mode update`에서 `SNS_AUTH_REQUIRED login_required`가 발생했다.
- 첫 공식 실행은 Windows 콘솔 인코딩 문제로 `UnicodeEncodeError: cp949`가 발생했으나, `PYTHONIOENCODING=utf-8` 지정 후 인증 실패 신호까지 도달했다.
- 2026-05-05 11:10 KST 기준, 일반 사용자가 여는 기본 Chrome에서는 `https://www.linkedin.com/my-items/saved-posts/`가 직접 열렸다. 확인 신호는 Chrome 창 제목 `(1) 저장한 게시물 | LinkedIn - Chrome`이다.

### 비파괴 probe 결과

- 기존 `storage_state.json`을 재사용하는 접근 방식은 모두 실패했다.
- 실패한 방식: 기본 context, `--web` 유사 viewport, `scrap_sns` 수집기 일치 환경, `sns_insight_update` 일치 환경, 실제 Chrome 채널, 임시 persistent profile에 기존 쿠키 주입.
- 따라서 "기존 인증 파일에 user_agent/locale/viewport만 다르게 줘서 되살린다"는 가설은 현재까지의 테스트에서는 지지되지 않는다.
- 추가 테스트로 `/login`의 `로그인중` 계정 카드/로그인 요소를 자동 클릭하는 fallback을 검증했다.
- 해당 fallback은 중간 UI 감지와 클릭 자체는 성공했지만, 이후 `checkpoint/challenge`의 reCAPTCHA 보안 인증으로 이동했고 saved-posts/recent-activity 통과에는 실패했다.
- 사용자가 관찰한 checkpoint 화면에는 "간단한 보안 인증"과 reCAPTCHA `로봇이 아닙니다` 체크박스가 표시됐다.
- 새 `storage_state` 생성과 LinkedIn 전용 임시 persistent profile 생성도 saved-posts 기준으로는 실패했다. 둘 다 recent-activity는 접근됐지만 saved-posts는 `login`으로 이동했다.
- 기존 `renew.py linkedin` interactive 방식도 가운데 로그인 버튼 클릭 후 `간단한 보안인증` 빈 페이지로 이동해 실패했다. 이 상태는 인증 성공이 아니므로 Enter 저장을 하지 않았다.
- 일반 Chrome 기본 프로필은 saved-posts 접근에 성공했으므로, 현재 문제는 LinkedIn 계정 또는 서버가 saved-posts 자체를 전면 차단한 상태라기보다 Playwright `storage_state` 또는 Playwright 기반 프로필의 신뢰 세션 문제로 보는 것이 더 타당하다.
- 사용자는 수동 export 기반 수집 MVP를 현재 범위에서 제외했다.
- 현재 실행 중인 일반 Chrome은 CDP endpoint로 붙을 수 없는 상태였다. `9222`, `9223`은 Chrome DevTools 응답이 없었고, `9229`는 `wrangler` 응답이었다.
- Playwright `launch_persistent_context`로 기본 Chrome user data dir의 `Default` 프로필을 여는 테스트는 saved-posts 탐색 전에 `TargetClosedError`로 종료됐다. Chrome 종료 코드는 `exitCode=21`이었다.
- 해당 테스트 전후로 canonical `storage_state.json`의 수정 시각과 크기는 변경되지 않았다.
- 2026-03-25 CDP/Chrome DevTools MCP 검토 문서들에서도 기본 Chrome 프로필 remote debugging 연결 실패, `DevToolsActivePort` 미생성, 승인 팝업, stale 파일, background MCP 간섭, transport 끊김 문제가 기록돼 있었다.
- 따라서 CDP는 LinkedIn 저장글 자동 수집의 남은 기본 후보가 아니라, 명시 승인 후 제한적으로 재현 가능한지 확인하는 진단용 후보로만 남긴다.

## 진행 과정 요약

1. 사용자는 Threads와 LinkedIn 모두 수집을 요청했다.
2. Threads 수집은 성공했으나, LinkedIn은 preflight에서 인증 벽이 감지되어 중단됐다.
3. 인증 파일은 존재했고 주요 쿠키가 단순 만료된 상태는 아니었지만, LinkedIn은 자동 브라우저 세션을 `checkpoint/challenge` 또는 `authwall`로 보냈다.
4. 사용자는 LinkedIn 로그인 화면에서 "로그인중 / 이 페이지에 남아 있을 경우 로그인됩니다" 중간 화면과 짧은 시간 나타나는 계정 버튼을 관찰했다.
5. 수집기와 인증갱신기의 접근 방식이 다른 점이 발견됐다.
6. 세 도구 비교표와 과거/현재 인증 구조 비교표를 작성했다.
7. 인증 안내문구가 현재 구조와 완전히 맞지 않는다는 문제가 발견되어 백로그에 추가했다.
8. 기존 `storage_state.json`을 여러 방식으로 재사용하는 비파괴 테스트를 수행했고 모두 실패했다.
9. `/login` 중간 UI의 계정 카드 클릭 fallback을 테스트했고, 클릭은 됐지만 reCAPTCHA checkpoint로 이동해 실패했다.
10. 새 `storage_state` 생성과 LinkedIn 전용 임시 persistent profile 생성은 recent-activity 접근까지만 성공했고, saved-posts는 `login`으로 이동해 실패했다.
11. 기존 `renew.py linkedin` interactive 방식도 가운데 로그인 버튼 클릭 후 `간단한 보안인증` 빈 페이지로 이동해 실패했다.
12. 일반 사용자가 여는 기본 Chrome에서는 saved-posts가 직접 열렸다.
13. 사용자는 수동 export 기반 수집 MVP를 현재 범위에서 제외했다.
14. 현재 열린 일반 Chrome은 CDP endpoint로 붙을 수 없었다.
15. Playwright가 기본 Chrome `Default` 프로필을 직접 여는 테스트도 `TargetClosedError`로 실패했다.
16. 2026-03-25 CDP/Chrome DevTools MCP 검토 문서들에서도 기본 프로필 remote debugging 연결 실패와 UX/운영 리스크가 이미 기록돼 있었다.
17. 따라서 CDP는 남은 자동화 후보가 아니라 기본 경로 제외 대상으로 둔다. 사용자가 명시 승인할 때만 진단용으로 재검증한다.
18. T1 git diff 분석으로 부록 B의 "갱신기 user_agent/locale 부재" 가설이 git 확정 사실로 격상됐다. 2026-04-18 `b82d153` AUTH_HOME 중앙화 커밋이 회귀 시점이며, 2026-05-05 `855e4bf`가 수집기 쪽 `Chrome/122` UA만 보강하여 불일치를 키웠다.
19. 단, 시나리오 ②(갱신기 환경 정렬로 saved-posts 통과)에 대해 코덱스 세션 1이 `probe_linkedin_new_auth.py`로 새 storage_state를 만들어 시도했고 saved-posts → login으로 보고했다. 이 보고는 "검토 필요한 사전 결과"로만 보존하며, 본 SPEC은 코덱스 실험을 확정 결론으로 채택하지 않는다. 본 세션에서 직접 재현·검증할 가설로 둔다.
20. 본 세션 v3 PoC(`probe_linkedin_scenario2_v3.py`)에서 `Chrome/122 + ko-KR + 1280x1000` + 단일 탭 흐름으로 새 storage_state를 만들어 saved-posts URL kind 통과, 카드 20건 노출, 사용자 육안 확인 완료. 시나리오 ② 통과 확정.
21. v4 PoC(`probe_linkedin_scenario2_v4.py`)에서 v3 storage_state를 로드해 saved 단독 / recent 단독 / saved→recent 3개 격리 시나리오 모두 PASS. 재현성과 페이지 독립 접근 가능성 확인.
22. v5 PoC(`probe_linkedin_scenario2_v5.py`)에서 단일 탭 / 두 탭 패턴 비교도 둘 다 PASS. 새 탭 가설은 결정적 변수가 아닌 것으로 판정.
23. 따라서 코덱스 세션 1의 `probe_linkedin_new_auth.py` 실행 보고는 false negative로 확정. 검증 깊이 부족이 원인이며 SPEC 결론으로 채택하지 않는다.
24. 운영 `linkedin_scrap.py`는 `chromium.launch` → `new_context(storage_state)` → 단일 `new_page` → `goto(SAVED_URL)` 흐름이라 v2의 두 탭 패턴은 운영에 존재하지 않는다. 따라서 새 탭 가설의 잔여 의문은 운영에 영향이 없어 추가 검증 보류로 결정.
25. 진짜 원인: 갱신기와 수집기의 fingerprint(user_agent/locale/viewport) 불일치. 2026-04-18 AUTH_HOME 중앙화 시점에 갱신기에서 user_agent 명시가 누락된 것이 회귀 시점이며, 본 세션 PoC가 갱신기 환경을 수집기와 정렬하면 saved-posts가 통과됨을 확정.

- 부록 B의 "갱신기 user_agent 부재"는 이제 추정이 아닌 git 확정 사실이다.

### 코덱스 세션 1의 사전 보고 — 검토 필요

코덱스 세션 1은 `scripts/diagnostics/probe_linkedin_new_auth.py`(2026-05-05 작성)에서 `Chrome/122 + ko-KR + viewport`를 적용해 새 storage_state를 만들고, saved-posts → login으로 보고했다(부록 C 참조).

이 보고는 **본 SPEC의 결정 근거로 채택하지 않는다**. 이유:
- 검증 깊이 한계: 일반 Chrome 기준선 검증이 창 제목 polling 한 번으로 끝나는 등 통제 비교가 비대칭이었다.
- 인간 육안 확인 부재: saved-posts 페이지의 저장글 카드 본문 렌더링을 사용자가 직접 확인한 기록이 없다.
- 실패 원인 추적 부족: saved-posts → login 리다이렉트의 정확한 원인(쿠키 거부, fingerprint 거부, 페이지 step-up 등)이 분리되지 않았다.

따라서 시나리오 ②는 **본 세션에서 직접 재현·검증할 가설**로 둔다(`TODO-09`). 통과 시 `TODO-10`으로 코드 변경 plan을 작성한다. 본 세션 재현도 실패하면 안티 디텍션 라이브러리 도입(playwright-stealth 등) 또는 자동 수집 보류로 분기한다.
