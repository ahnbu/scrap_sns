# SNS 수집 결과 정합성 검증

## 목적

SNS 수집 이슈를 진단할 때 "플랫폼에는 보이는데 로컬 웹페이지에는 안 보인다", "수집은 됐는데 화면에서는 누락처럼 보인다"는 문제가 반복됐다. 앞으로는 수집 성공 여부를 스크래퍼 실행 결과만으로 판단하지 않고, 영구 저장소(DB/JSON 결과 파일)와 로컬 서버가 내려주는 화면용 데이터까지 같은지 확인한다.

이번 세션의 핵심 교훈은 수집 누락과 화면 표시 누락을 분리해야 한다는 점이다. LinkedIn 최신 글로 지목된 글은 실제로 수집 파일과 서버 응답에 있었고, 새 브라우저 상태의 웹페이지에서도 보였다. 따라서 후속 개선은 "수집 완료 후 정합성 검증을 강제"하는 방향이 우선이다.

## 요구사항

- R1. **수집 완료 판정 강화**: 수집 스크립트가 성공 종료해도, 수집 전후 통합 결과 파일을 비교해 만든 신규 글 샘플이 로컬 서버 응답에 반영됐는지 확인하기 전에는 완료로 보지 않는다.
- R2. **표시 누락과 수집 누락 분리**: 사용자가 "안 보인다"고 말한 글은 먼저 `platform_id`/URL 기준으로 플랫폼 파일, 통합 파일, 서버 API, 웹페이지 기본 화면을 순서대로 대조한다.
- R3. **정렬 기준 명시**: 게시글 작성시간, 로컬 수집시간, 플랫폼 저장시간을 섞어 쓰지 않는다. 현재 데이터에 플랫폼 저장시간이 없으면 "알 수 없음"으로 표시한다.
- R4. **로컬 서버 정합성 강제**: 수집 후 로컬 서버가 읽는 최신 통합 파일과 `/api/posts` 화면용 데이터가 맞는지 검증한다. `/api/search`는 기본 정합성 검사가 아니라 별도 진단이 필요할 때만 사용한다.
- R5. **브라우저 상태 영향 분리**: 검색어, 플랫폼 필터, 태그, 즐겨찾기, 할 일, 숨김 상태 때문에 안 보이는 경우를 수집 실패로 오판하지 않는다.
- R6. **가짜 문제 대응 방지**: 코드 변경이나 개선안 제시 전에 실제 실행 결과와 1차 데이터 근거를 확보한다.
- R7. **업데이트 후 화면 갱신 검증**: 업데이트 작업 완료 뒤 단순히 데이터를 다시 불러오는 것에 그치지 않고, 실제 화면 데이터가 새 서버 응답을 반영했는지 확인한다.
- R8. **정렬 용어 문서화**: `created_at`, `crawled_at`, `sequence_id`, `platform_sequence_id`, `platform_saved_at`의 의미를 README에 명확히 설명한다.

## 성공 기준

- S1. 특정 글 ID 또는 URL을 제시하면 플랫폼 결과 파일, `output_total/total_full_*.json`, `/api/posts` 중 어디에 있고 어디에 없는지 한 번에 판정할 수 있다. 검색 진단이 필요하면 `/api/search`를 추가로 확인한다.
- S2. 수집 완료 판정은 "스크래퍼 성공"과 "신규 글 샘플의 로컬 서버 반영"을 구분한다.
- S3. 정합성 검증이 정상 통과하면 추가 메시지를 노출하지 않는다. 실패 또는 3초 초과 시에만 성공 메시지를 차단하고 확인 필요 상태를 표시한다.
- S4. LinkedIn 예시 글 `7456682555220680704` 같은 케이스는 "수집 누락"이 아니라 "화면 상태 확인 필요"로 분류된다.
- S5. 사용자가 보는 웹페이지 데이터 상태에는 서버가 지정한 신규 샘플 글이 포함된다.
- S6. 업데이트 완료 후 화면이 새 데이터를 반영하지 못하면 "업데이트 성공"만 표시하지 않고, "화면 갱신 확인 실패"를 별도로 표시한다.
- S7. README만 보고도 "작성일순", "로컬 수집순", "플랫폼 저장순"의 차이를 이해할 수 있다.

## 제약 조건

- 플랫폼 저장시간은 현재 LinkedIn/Threads 데이터에 별도 필드로 저장되지 않는다. 현재 확인 가능한 값은 게시글 작성시간과 로컬 수집시간이다.
- 로컬 웹페이지는 브라우저의 개인 상태를 가진다. 검색어, 필터, 숨김, 태그, 즐겨찾기, 할 일 상태가 결과 표시를 바꿀 수 있다.
- 서버는 최신 `output_total/total_full_*.json`을 읽어 `/api/posts`와 `/api/search`를 제공한다. 따라서 플랫폼 파일만 갱신되고 통합 파일이 갱신되지 않으면 웹페이지와 어긋난다.
- 현재 웹페이지는 업데이트 성공 후 `fetchData()`를 호출하고, 서버가 내려준 플랫폼별 신규 샘플이 새 `/api/posts` 결과에 들어왔는지 확인한다.
- 기존 작업 세션에서 X 로그인 패널 UX 관련 변경이 진행 중이므로, 이 SPEC은 수집/정합성 진단 범위에 한정한다.

## 결정안

트레이드오프: 검증 단계가 늘어나 수집 완료까지 시간이 조금 더 걸리지만, 같은 정합성 문제를 반복 조사하는 비용이 더 크다.

### D1. 수집 완료의 정의를 바꾼다

수집 완료는 "스크래퍼 실행 성공"이 아니라 다음 4단계가 모두 맞는 상태로 정의한다.

1. 플랫폼별 결과 파일에 글이 있다.
2. 통합 결과 파일에 같은 글이 있다.
3. 로컬 서버 API가 같은 글을 내려준다.
4. 웹페이지가 다시 불러온 화면용 데이터에 같은 글이 있다.

### D2. 핵심 식별자는 URL보다 `platform_id`를 우선한다

LinkedIn은 URL query가 길고, Threads는 `threads.net`/`threads.com` 차이가 생길 수 있다. 따라서 정합성 검증은 `platform_id` 또는 게시글 code를 1순위로 쓰고, canonical URL과 본문 검색은 보조 기준으로 사용한다.

### D3. 정렬명은 사용자가 이해하는 기준으로 바꾼다

현재 "최근순"은 게시글 작성시간 기준이다. "저장순"은 실제 플랫폼 저장시간이 아니라 `sequence_id` 순서이므로 이름과 설명이 오해를 만든다. 후속 계획에서는 "작성일순", "로컬 수집순", "플랫폼 저장순(필드 확보 시)"으로 분리한다.

### D4. 정합성 실패 시 단계별 원인을 보여준다

예: 수집 전후 차이에는 있으나 `/api/posts` 재조회 결과에 없으면 서버 반영 문제로 분류한다. 정상 통과 시에는 별도 메시지를 보여주지 않고, 누락 또는 timeout일 때만 수집 결과 모달에 표시한다.

### D5. 업데이트 후 화면 갱신은 "시도"와 "검증"을 분리한다

현재 구현은 업데이트 성공 후 데이터를 다시 불러오고, 새 통합 파일의 신규 샘플이 서버 API 결과에 실제로 반영됐는지 검증한다. 정상 통과 시에는 정합성 메시지를 숨기고, 누락 또는 timeout일 때만 업데이트 확인 필요 상태를 보여준다.

### D6. README에 시간·정렬 개념을 문서화한다

정렬 용어는 사용자 오해를 줄이기 위해 README에 먼저 고정한다. 현재 기준의 권장 정의는 다음과 같다.

- `created_at`: 게시글이 플랫폼에 작성된 시간
- `crawled_at`: 내 로컬에 처음 수집된 시간
- `sequence_id`: 현재 로컬 통합 목록의 순서 번호
- `platform_sequence_id`: 플랫폼별 수집 목록에서의 순서
- `platform_saved_at`: 실제 내가 저장한 시간. 현재는 수집되지 않음

따라서 현 상태에서 "저장순"은 실제 저장시간순이 아니라 "로컬 수집시간과 플랫폼 내 순서를 섞어 만든 순서"로 설명해야 한다.

### D7. 정합성 확인 계획

업데이트 완료 후 서버가 수집 전후 `total_full_YYYYMMDD.json`을 비교해 플랫폼별 신규 샘플을 만든다. 화면은 수집 후 이미 다시 불러온 `/api/posts` 결과 안에 이 샘플들이 있는지만 확인한다.

1. 수집 전 최신 통합 파일의 `platform_id` 또는 canonical URL 키를 기억한다.
2. 수집 후 최신 통합 파일에서 새로 생긴 글을 플랫폼별로 분리한다.
3. LinkedIn, Threads, X 각각 신규 글 최대 3개를 `new_samples`로 응답한다. 3개 미만이면 있는 만큼만 응답한다.
4. 웹페이지는 추가 검색 호출 없이 `/api/posts` 재조회 결과에서 샘플 키가 모두 있는지 확인한다.
5. 정상 통과 시 정합성 메시지는 숨긴다.
6. 누락 또는 3초 초과 시에만 `업데이트 확인 필요`로 표시한다.

### D8. 기존 수집 결과 모달은 실패 표시만 재사용한다

최근 커밋 `f7205f4 feat(viewer): 수집 결과 모달과 인증 안내 추가`에서 수집 결과 모달, 플랫폼별 신규 건수, 수집 실패, 인증 갱신 안내가 추가됐다. 이 모달은 정합성 실패 또는 timeout 표시 위치로 재사용한다. 정상 통과 시에는 `정합성 확인` 섹션을 노출하지 않는다.

### D9. 실제 플랫폼 저장시간은 백로그 후보로 둔다

현재 파일과 스냅샷에서는 실제 저장시간 필드가 확인되지 않았다. 이 세션에서 더 진행하려면 LinkedIn/Threads 원본 요청을 추가로 추적하는 별도 탐색이 필요하다. 정합성 개선의 핵심 경로를 막는 항목은 아니므로, 별도 백로그 후보로 분리하는 것을 기본 권장안으로 둔다.

### D10. 구현 결과

커밋 `83d2617 feat(consistency): 신규 수집 샘플 정합성 확인 추가`로 반영했다.

- 서버: `/api/run-scrap` 응답에 `new_counts`, `new_samples`를 포함한다.
- 화면: `fetchData()` 이후 `new_samples`가 `/api/posts` 결과에 있는지 메모리에서 비교한다.
- 성능: 기본 검사에서 `/api/search` 호출을 제거했다.
- 제한시간: 정합성 확인은 최대 3초까지 기다린다.
- 표시정책: 통과 시 숨김, 실패/timeout 시 `업데이트 확인 필요`와 단계별 누락 정보를 표시한다.

- F8. Threads는 이전 진단에서 신규로 보인 글 코드 14개가 플랫폼 파일, 통합 파일, 서버 API에 없었다. 다만 사용자가 말한 "쓰레드는 다 들어가 있다"와 기준이 다를 수 있어 후속에서는 글 코드/URL 기준으로 다시 맞춰야 한다.

- F14. 최근 커밋 `f7205f4`는 `index.html`, `web_viewer/script.js`, `web_viewer/style.css`, `tests/unit/test_web_viewer_scrap_result_helpers.py`를 중심으로 수집 결과 모달을 추가했다.

## 부록: 정합성 검증 체크리스트

수집 완료 직후 아래 체크를 통과해야 한다.

1. 수집 전후 최신 통합 파일을 비교해 신규 글 키를 만든다.
2. LinkedIn, Threads, X별 신규 글 최대 3개를 샘플로 잡는다.
3. 로컬 서버 `/api/posts` 재조회 결과에 같은 `platform_id` 또는 canonical URL 키가 있다.
4. 샘플이 모두 있으면 정합성 메시지를 숨기고 업데이트 완료로 본다.
5. 샘플이 누락되면 `업데이트 확인 필요`로 보고 어느 플랫폼 샘플이 몇 개 누락됐는지 표시한다.
6. 3초 안에 확인이 끝나지 않으면 timeout으로 보고 `업데이트 확인 필요`로 표시한다.
