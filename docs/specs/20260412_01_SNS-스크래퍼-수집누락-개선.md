# SNS 스크래퍼 수집 누락 개선

## 목적

웹 뷰어에서 "업데이트" 버튼 클릭 후 수집 결과를 확인했을 때, 실제 SNS에서 저장한 게시물이 누락되는 문제를 해결한다. 플랫폼별로 누락 원인이 다르므로 각각 대응한다.

## 요구사항

- R1. **로그 누적 보존**: 스크래핑 로그가 매 실행마다 덮어쓰기(`"w"`)되어 이전 이력이 사라지는 문제를 해결한다 → `"a"` (append) 모드로 변경 ✅ 완료
- R2. **Threads update 모드 누락 수정**: 최근 저장했지만 게시물 작성일이 오래된 경우, 기준점보다 아래에 위치하여 수집되지 않는 구조적 결함을 해결한다
- R3. **Twitter 본문 오염 현황 기록**: X의 봇 차단 리다이렉트로 인해 원래 게시물이 아닌 다른 페이지의 본문이 수집되는 문제를 기록하고, 추후 과제로 이관한다
- R4. **LinkedIn 뷰어 표시 확인**: 수집은 정상이지만 웹 뷰어에서 검색/표시가 안 되는 경우가 있는지 확인하고, 재현 조건을 파악한다

## 성공 기준

- R1: 로그 파일이 실행마다 누적되며, 구분선으로 실행별 식별 가능
- R2: Threads에서 누락 8건(신규 5건 + 기존 누락 3건)이 update 모드로 수집됨. 신규 5건: `DQ8mqRaEtUL`, `DW7wRVUD82s`, `DW8k05MEj9D`, `DVVRaH3kYkF`, `DWq9TgCGcp3`. 기존 누락 3건: `DWn9J1mEpb4`, `DWROO2NDQ2i`, `DWeNl73jxfT`
- R3: BACKLOG.md에 Twitter 본문 오염 이슈가 등록됨 (session ID 포함)
- R4: ~~LinkedIn 뷰어 표시 확인~~ → 이슈 없음으로 닫음. 스크래퍼 정상, 뷰어 기본 정렬(`date`=`created_at`)에서 5번째·9번째로 정상 표시됨. 코드 변경 불필요.

## 제약 조건

- Threads 저장 목록의 정렬 순서는 **게시물 작성일순**이며, 저장 시점순이 아님 (플랫폼이 결정, 변경 불가)
- Twitter/X는 봇 차단이 강해 0.5~1초 내 리다이렉트 발생 — 스크래퍼 로직만으로 해결 어려움
- 기존 3층 스키마 게이트(Threads)를 유지해야 함

### D2. Threads update 모드 누락 해결 (R2)

**근본 원인**: `thread_scrap.py`의 update 모드가 기준점(stop_codes) 발견 시 즉시 중단하는데, Threads 저장 목록이 게시물 작성일순으로 정렬되어, 최근 저장했지만 오래된 게시물이 기준점 아래에 위치 → 영원히 수집 불가.

**현재 코드 흐름** (`thread_scrap.py`):
- 라인 249-253: `posts[:5]`에서 stop_codes 설정 (sequence_id 역순, 즉 최근 저장 5건)
- 라인 503-507: DOM 스캔에서 기준점 발견 시 `break` (즉시 중단)
- 라인 367-371: 네트워크 응답에서 기준점 발견 시 `return` (즉시 중단)
- 라인 584-587: 스크롤 루프에서 `stop_code_found` 확인 시 `break`

**기존 중복 방지 현황** (코드 분석 확인):
- `all_posts_map` (dict, 라인 225·240): 기존 전체 665건을 `code → post` 로 메모리에 로드. 이미 존재함.
- 라인 374-375, 522-523: 현재 세션 내 `collected_data`에서만 중복 방지. **기존 DB(`all_posts_map`)와의 비교는 수집 중 수행하지 않음.**
- 결론: `all_posts_map`을 중단 조건으로 활용하면 최소 변경으로 해결 가능.

**채택 접근법: 기수집 연속 카운터 방식**

기준점(stop_codes) 즉시 중단을 폐기하고, 아래로 교체:

```
기존 수집 set: all_posts_map의 key set (665개)
스크롤하며 게시물 수집 시:
  code가 set에 없음 → 신규, 수집 + 연속카운터 리셋(0)
  code가 set에 있음 → 기수집, 연속카운터 +1
  연속카운터 >= N (예: 10) → 중단
```

트레이드오프: 기존보다 스크롤이 약간 늘어날 수 있으나, 최대 50회 제한이 이미 존재하므로 무한 스크롤 위험 없음. **N=20** — 실측에서 신규 5건 이후 기수집 16건 연속 구간이 확인됨. N=20이면 이 구간을 통과하고 누락 3건까지 도달 가능.

**변경 지점** (`thread_scrap.py`):
- 라인 226: `stop_codes = []` → `existing_codes = set(all_posts_map.keys())`, `consecutive_dup_count = 0` 추가
- 라인 249-253: stop_codes 설정 블록 제거
- 라인 367-371 (네트워크): stop_codes 체크 → 기수집 카운터 로직으로 교체
- 라인 503-507 (DOM): 동일
- 라인 584-587 (스크롤): stop_code_found → consecutive_dup_count >= N 으로 교체

- **F3**: Twitter 2건(`2039805659525644595`, `2038713289254064321`)은 platform_id로는 존재하나, full_text가 "몇년 전만 해도 아마존 본사에서 풀타"로 동일하게 오염됨. 원래 본문("LLM Knowledge Bases", "This guy literally broke down")과 불일치.

- **F5**: 로그 파일(`logs/threads.log`)에서 "기준 게시물 발견! (code: DW9LwQTiYNi) - DOM 스캔 중단" 확인 — 스크롤 1회차에서 즉시 중단됨.
- **F6**: `thread_scrap.py` 코드에 grace period, 작성일/저장일 구분, 기준점 이후 계속 탐색 등의 안전장치 없음 확인.
- **F7**: `all_posts_map` dict (라인 225·240)에 기존 전체 데이터 665건이 이미 로드됨. 수집 중 이 dict와의 비교는 현재 수행하지 않으나, 중단 조건으로 즉시 활용 가능.
- **F8**: sequence_id(저장 순서)와 created_at(게시물 작성일)은 완전히 다른 순서임 확인. 예: seq=651이 seq=654보다 created_at이 최신(4/11 12:34 vs 4/11 00:21).
- **F11**: LinkedIn 2건은 뷰어에서 정상 표시됨. 기본 정렬(`date`)에서 created_at 기준 5번째·9번째. sequence_id 정렬에서는 313·314번째(중간)로 묻힐 수 있으나 date 정렬이 기본이므로 문제 아님. 스크래퍼·뷰어 모두 정상.
- **F9**: Playwright로 실제 Threads 저장 목록 확인 (2026-04-12). 목록 순서: [1~4] 기수집(기준점 영역) → [5~9] 🆕신규 5건 → [10~25] 기수집. 누락 3건(`DWn9J1mEpb4`, `DWROO2NDQ2i`, `DWeNl73jxfT`)은 25건 이후 더 아래에 위치. 기준점([1]) 직후 신규가 나오므로 N=1이어도 이 케이스는 통과하나, 누락 3건까지 도달하려면 기수집 구간 최대 ~20건을 통과해야 함 → **N=20 권장**.
- **F10**: 이번 update 실행(16:32)에서 🆕신규 5건(`DQ8mqRaEtUL`, `DW7wRVUD82s`, `DW8k05MEj9D`, `DVVRaH3kYkF`, `DWq9TgCGcp3`)도 기준점 아래에 있어 수집되지 않았을 가능성 있음 — 별도 확인 필요.
