# 스크래퍼 TDD(Test-Driven Development) 활용 가이드 (2026-03-10)

> 본 문서는 SNS 프로젝트의 유지보수 효율을 높이기 위해 도입된 TDD 체계의 구체적인 활용 방법을 설명합니다.

---

## 1. 왜 TDD를 사용해야 하나요? (도입 배경)
- **브라우저 로딩 대기 제거**: 매번 30초~1분씩 걸리는 브라우저 실행 없이 1초 내에 로직 검증 가능.
- **계정 차단 예방**: SNS 사이트 반복 접속을 줄여 봇 감지 위험 최소화.
- **코드 안정성**: 하나의 기능을 고쳤을 때 다른 플랫폼이나 기존 필드가 깨지는 것을 즉시 발견.

---

## 2. 실전 활용 시나리오

### 상황 A: SNS 사이트 구조가 변경되어 스크랩이 실패할 때
1. **증거 확보**: `tests/fixtures/snapshots/{platform}/` 폴더에서 가장 최근에 실패한 시점의 HTML/JSON 파일을 확인합니다.
2. **테스트 데이터 교체**: 위에서 찾은 파일을 `tests/fixtures/{platform}_sample.html`로 복사(덮어쓰기)합니다.
3. **실패 확인**: 터미널에 명령어를 입력하여 현재 코드가 실패함을 확인합니다.
   ```bash
   pytest tests/unit/test_{platform}_parser.py
   ```
4. **로직 수정**: `utils/{platform}_parser.py`에서 바뀐 태그나 JSON 경로를 수정합니다.
5. **무한 반복**: `pytest`를 다시 실행하여 초록색(PASSED)이 뜰 때까지 4번 과정을 반복합니다. (브라우저는 켜지 마세요!)
6. **최종 승인**: 테스트가 통과되면 실제 스크래퍼를 돌려도 안전합니다.

### 상황 B: "게시물 좋아요 수" 같은 새 필드를 추가하고 싶을 때
1. **명세 작성**: `tests/unit/test_{platform}_parser.py`의 테스트 함수 끝에 기대값을 추가합니다.
   ```python
   assert post["likes_count"] > 0
   ```
2. **실패 확인**: `pytest` 실행 시 `KeyError`나 `AssertionError`가 뜨는 것을 확인합니다.
3. **기능 구현**: `utils/{platform}_parser.py`에서 해당 값을 추출하는 코드를 추가합니다.
4. **성공 확인**: 다시 `pytest`를 실행하여 통과함을 확인합니다.

---

## 3. 주요 명령어
- **전체 파서 테스트 실행**: `pytest tests/unit/`
- **특정 플랫폼만 테스트**: `pytest tests/unit/test_linkedin_parser.py`
- **상세 출력 포함**: `pytest -v`

---

## 4. 스냅샷 관리 유의사항
- `utils/common.py`의 `save_debug_snapshot()` 함수는 용량 관리를 위해 플랫폼당 **최신 10개**의 스냅샷만 유지합니다.
- 중요한 테스트 케이스가 포함된 데이터라면 `tests/fixtures/` 폴더로 옮겨서 영구적으로 관리하세요.
