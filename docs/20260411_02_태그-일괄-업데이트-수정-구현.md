- `migrateLegacyTagKeys(posts)` helper를 추가해 `undefined`, `code-only`, `https://www.threads.net/t/<code>`를 canonical URL로 병합하게 했다.
- `fetchData` 전처리 직후 `migrateLegacyTagKeys(allPosts)`를 1회 실행하도록 연결했다.

문서화 시점 기준 plan 변천 이력은 없었다. `find_plan_versions.js` 결과는 `{"versions":[]}`였다.

### 변경 3: `fetchData` 내 1회 복구 블록 — regex heuristic 폐기 → canonical map 방식

**Codex 권장안 채택**: 휴리스틱 정규식 대신 `allPosts`에서 직접 canonical URL map을 구성하고, stale 키를 3종 모두 이동.

삽입 위치: `web_viewer/script.js:471` 직전 (`allPosts` 전처리 이후 + `applyAutoTags(..., true)` 이전).

```js
// One-time repair: migrate legacy tag keys → canonical URLs
{
    // Build canonical URL lookup from current posts
    const canonByCode = new Map();      // code/platform_id → canonical full URL
    const canonByTSlash = new Map();    // "https://www.threads.net/t/<code>" → canonical full URL
    for (const p of allPosts) {
        const canonical = resolvePostUrl(p);
        if (!canonical) continue;
        const code = p.platform_id || p.code;
        if (code) {
            canonByCode.set(code, canonical);
            canonByTSlash.set(`https://www.threads.net/t/${code}`, canonical);
        }
    }

    let migrated = 0;
    const legacyKeys = Object.keys(postTags);
    for (const key of legacyKeys) {
        if (key === 'undefined') {
            delete postTags[key];
            migrated++;
            continue;
        }
        let target = null;
        if (canonByTSlash.has(key)) {
            target = canonByTSlash.get(key);
        } else if (canonByCode.has(key)) {
            target = canonByCode.get(key);
        }
        if (target && target !== key) {
            const merged = new Set([...(postTags[target] || []), ...(postTags[key] || [])]);
            postTags[target] = Array.from(merged).filter(t => t && t.trim().length > 0);
            delete postTags[key];
            migrated++;
        }
    }

    if (migrated > 0) {
        console.log(`🔧 Migrated ${migrated} legacy tag keys to canonical URLs`);
        localStorage.setItem('sns_tags', JSON.stringify(postTags));
        syncTagsToServer();
    }
}
```

**이 방식의 장점**:
- 휴리스틱 길이 제한 없음 → 39자 `DUniriOjLVh0U...` code도 정상 복구.
- `/t/<code>` 네임스페이스 167건과 `code-only` 659건을 한 루프에서 처리.
- Threads 플랫폼 체크 불필요: canonical map 자체가 `allPosts`에서 생성되므로 LinkedIn/X의 19자리 `platform_id`도 이론상 포함되지만, 그들의 canonical URL은 `https://www.linkedin.com/...` → 기존에 `linkedin http` 네임스페이스(101건)에 저장된 키와 매칭되지 않으므로 이동 없음. 오이동 가능성은 **LinkedIn/X의 `platform_id`와 동일한 code-only 키가 우연히 존재하는 경우**만이며, 이는 현재 서버 JSON의 `code-only` 659건이 전부 Threads만 있다면 0건, 섞여 있다면 "어느 쪽 canonical로 이동해도 같은 포스트"이므로 안전.
- 관리 비용: 휴리스틱 정규식 유지 불필요.

**미커버 영역**: `other` 172건은 위 3종(`undefined`/`/t/`/`code-only`) 분류에 들지 않는 기타 키. 기존 정상 포맷(LinkedIn URL 등) 혹은 식별 불가 stale. 이 범주는 건드리지 않고 보존.
