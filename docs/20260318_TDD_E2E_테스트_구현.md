### 0. CORS 전체 허용 롤백 (사용성 트레이드오프 해소)

**server.py 수정** (테스트 작성 전 선행):
```python
# Before

# After
CORS(app)
```

**근거**: `null` origin만 추가하는 것은 "보안을 지키는 척하면서 실질적으로 안 지키는" 중간 지대. 로컬 전용 도구에서는 전체 허용이 더 솔직하고 깔끔하다. CORS 테스트(S11, S12)는 제외.

### 1. Flask test client vs 실제 HTTP 요청

| | Flask test client | requests + 실제 서버 |
|---|---|---|
| 서버 기동 필요 | 불필요 | 필요 |
| CI 안정성 | 높음 | 낮음 (포트 충돌, 타임아웃) |
| 보안 로직 검증 | 완전 (WSGI 레벨) | 완전 |
| **채택** | **S1~S11 보안 테스트** | U1~U5 UI 테스트 (Playwright 필요) |

**근거**: 보안 테스트는 API 라우팅·입력 검증 로직이 핵심이므로 test client로 충분. UI 테스트만 실제 서버+Playwright 사용.

### 2. XSS 테스트 방법

| | fixture JSON 파일 | page.evaluate 주입 |
|---|---|---|
| 설정 복잡도 | 높음 (data.js 교체 필요) | 낮음 |
| 실제 렌더링 경로 | 우회 | 실제 createCard() 경유 |
| **채택** | — | **page.evaluate로 allPosts에 주입 후 renderPosts() 호출** |

**근거**: data.js는 정적 파일이라 테스트 시 교체가 번거롭다. `page.evaluate()`로 JS 객체에 직접 XSS payload를 주입하면 실제 렌더링 파이프라인(createCard → escapeHtml)을 그대로 타므로 더 정확한 검증이 가능하다. 별도 fixture 파일(`xss_payload_posts.json`)은 payload 패턴 참조용으로만 유지.

### 3. save-tags 부작용 관리

`POST /api/save-tags`는 실제 파일(`web_viewer/sns_tags.json`)을 쓴다. 테스트에서 `server.WEB_VIEWER_DIR`을 `tmp_path`로 교체하여 원본 파일 보호.

### 4. integration 폴더 신설 이유

기존 unit 테스트는 개별 함수 단위, e2e는 브라우저/API 단위. 파서 통합 테스트는 "fixture → 파서 체인 → 표준 스키마"를 검증하므로 중간 계층이 필요. 기존 unit 테스트와 중복 방지를 위해 **스키마 준수 + 필드 완전성**에 집중.
