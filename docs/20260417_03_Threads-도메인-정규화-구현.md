# Threads 도메인 정규화 구현

## 발단: 사용자 요청

사용자는 Threads 도메인을 `threads.net`에서 `threads.com`으로 정규화하고, 관련 태그·상태·데이터·테스트까지 일관되게 전환하라고 요청했다.

직전 검토에서 `sns_tags`만이 아니라 `favorites`, `todos`, `hidden`, `folded` 같은 URL-key 상태와 `query-sns.mjs` 조회 경로까지 함께 다뤄야 회귀를 막을 수 있음이 확인됐다.

## 작업 상세내역

- 기존 코드와 실데이터를 먼저 검토해 `.net` 하드코딩 위치, 태그 저장 구조, 출력 JSON의 URL 상태를 확인했다.
- 실측 결과 `web_viewer/sns_tags.json`에는 `.net`/`.com`/`/t/{code}`가 혼재했고, 최신 `output_total/total_full_20260417.json`의 Threads URL은 전부 `.net`이었다.
- 구현 범위는 다음으로 고정했다.
  - 뷰어 canonical resolver와 기존 `migrateLegacyTagKeys(posts)` 확장
  - `query-sns.mjs` resolver 동기화
  - Python Threads write/navigation URL `.com` 전환
  - file-based migration script 추가
  - 관련 unit/smoke test 기대값과 신규 테스트 보강
- 새 one-shot migrator는 만들지 않고, 기존 migrator를 확장해 URL-key 상태 전체를 같은 canonical map으로 마이그레이션한다.

## 의사결정 기록

| 항목 | 호환 우선 | 일괄 전환 | 뷰어 우선 |
|------|-----------|-----------|-----------|
| 기존 태그/상태 보존 | ✅ 가장 유리 | ⚠️ 별도 즉시 마이그레이션 필수 | ⚠️ 일부만 해결 |
| CLI/데이터 혼재 대응 | ✅ 가능 | ⚠️ 강한 순서 의존 | ❌ 깨질 위험 큼 |
| 구현 복잡도 | 보통 | ⚠️ 높음 | ✅ 낮음 |
| 회귀 위험 | ✅ 가장 낮음 | ❌ 높음 | ❌ 높음 |

<span style="color:#888">*정렬 기준: 사용자 데이터 손실 방지와 실제 운영 경로 회귀 방지를 최우선으로 두었다.*</span>

- 결정: `.com` canonical + 호환 우선
- 근거: 실데이터가 아직 `.net` 중심이고, URL-key localStorage/CLI/file outputs가 동시에 얽혀 있어 단번 전환은 회귀 가능성이 높다.
- 트레이드오프: read-side 호환 코드와 migration 검증이 늘어나 구현량은 증가하지만, 사용자 상태 유실을 피할 수 있다.
