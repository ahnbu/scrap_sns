# 분석 보고서: Threads 스크래핑 실행 방식에 따른 동작 차이 규명

`total_scrap.py`를 통한 전체 실행과 `threads_scrap.py` 단독 실행 시 발생하는 로그 및 동작 차이를 분석한 결과, 시스템의 'Producer-Consumer' 구조에 의한 정상적인 현상임을 확인했습니다.

## 1. 핵심 원인: 단계별 역할 분담
본 시스템은 효율적인 데이터 수집을 위해 2단계 구조를 채택하고 있습니다.

- **Phase 1 (Producer)**: `threads_scrap.py`
    - 역할: Threads '저장됨' 목록을 빠르게 스캔하여 게시물 코드(ID)와 기본 정보만 수집.
    - 결과물: `threads_py_simple_full_*.json` (목록형 데이터)
- **Phase 2 (Consumer)**: `scrap_single_post.py`
    - 역할: Phase 1에서 수집된 ID들을 바탕으로 개별 페이지에 접속하여 본문, 타래글, 이미지를 정밀 수집.
    - 결과물: `threads_py_full_*.json` (상세 병합 데이터)

## 2. 실행 방식별 로그 차이 분석

### A. `threads_scrap.py --mode update` 단독 실행 시
- **동작**: 오직 Phase 1(목록 스캔)만 수행합니다.
- **로그 의미**: "수집된 데이터가 없습니다"라는 메시지는 **'목록 상에 아직 Full DB로 넘어가지 않은 신규 항목이 없다'**는 뜻이 아니라, **'목록 스캔 결과 기준 게시물(Stop Code) 이후의 새로운 URL이 발견되지 않았다'**는 의미입니다. 
- **결과**: 목록만 업데이트하고 종료되므로 매우 짧은 시간(약 11초)이 소요됩니다.

### B. `total_scrap.py --mode update` 실행 시
- **동작**: `threads_scrap.py` 실행 후 곧바로 `scrap_single_post.py`를 호출합니다.
- **로그 의미**: "Simple에서 20개의 신규 데이터를 Full DB로 가져왔습니다"라는 메시지는 `scrap_single_post.py`가 시작되면서 **Simple DB(목록)에는 있지만 아직 상세 수집(Full DB)이 안 된 항목들**을 식별하여 작업을 시작했음을 알리는 것입니다.
- **결과**: 개별 게시물 상세 수집 작업이 수반되므로 수집 건수에 따라 실행 시간이 길어집니다.

## 3. 결론 및 권장 사항
- **오해**: 단독 실행 시 데이터가 누락되는 것이 아니라, 단독 실행은 '목록 확보'까지만 담당하는 것입니다.
- **권장**: 최신 데이터를 완벽하게 웹 뷰어까지 반영하려면 항상 `total_scrap.py`를 실행하거나, `threads_scrap.py` 실행 후 반드시 `scrap_single_post.py`를 수동으로 실행해 주어야 합니다.
- **향후 개선 제안**: 단독 실행 시에도 사용자에게 "다음 단계(상세 수집)가 필요함"을 알리는 가이드 메시지를 출력하도록 개선할 수 있습니다.
