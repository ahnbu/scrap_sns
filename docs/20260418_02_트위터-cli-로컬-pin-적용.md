# twitter-cli 로컬 pin 적용 및 검증

## 발단: 사용자 요청

사용자는 "이 레포에서 쓰는 것만 `twitter-cli` 버전을 `0.8.5`로 pin할 수 있는가?"를 물었고, 이어서 `doc-act`로 문서를 만든 뒤 실행 결과까지 같은 문서에 반영하라고 요청했다.

문제의 핵심은 현재 consumer가 PATH의 전역 `twitter` 실행 파일에 의존한다는 점이다. 이 상태에서는 `pipx upgrade twitter-cli` 같은 사용자 전역 업데이트가 이 레포 실행 결과에 직접 영향을 준다.

## 작업 상세내역

현재 구조를 확인한 결과, `requirements.txt`에는 `twitter-cli`가 없고 [`utils/twitter_cli_adapter.py`](../utils/twitter_cli_adapter.py) 는 `["twitter", "tweet", url, "--json"]` 형태로 PATH의 전역 CLI를 호출하고 있었다.

이를 해결하는 방법은 세 가지였다.

| 항목 | 전역 pipx 유지 | 레포 로컬 venv + bare `twitter` | 레포 로컬 venv + `sys.executable -m twitter_cli.cli` |
|------|---|---|---|
| 레포 단위 버전 고정 | ❌ 불가 | ⚠️ PATH 영향 남음 | ✅ 가능 |
| 전역 업그레이드 영향 차단 | ❌ 불가 | ⚠️ 환경 설정 의존 | ✅ 차단 |
| 구현 복잡도 | ✅ 가장 낮음 | 보통 | 보통 |
| 운영 안정성 | ❌ 낮음 | ⚠️ 조건부 | ✅ 가장 높음 |

<span style="color:#888">*정렬 기준: 이 작업의 핵심은 "이 레포만 별도 버전 고정"이므로, 전역 영향 차단과 재현성이 가장 중요하다.*</span>

실행 단계에서는 세 파일만 바꿨다. `requirements.txt`에 `twitter-cli==0.8.5`를 추가했고, `utils/twitter_cli_adapter.py`의 호출을 `sys.executable -m twitter_cli.cli`로 바꿨다. 그리고 `tests/unit/test_twitter_cli_adapter.py`에 새 호출 경로를 검증하는 failing test를 먼저 넣고, 그다음 구현을 맞췄다.

## 의사결정 기록

- 결정: `requirements.txt`에 `twitter-cli==0.8.5`를 추가하고, consumer 호출을 `sys.executable -m twitter_cli.cli` 기반으로 바꾼다.
- 근거: 레포가 사용하는 Python 환경 안에서만 `twitter-cli` 버전을 고정할 수 있고, 전역 `pipx` 업데이트와 PATH 변동 영향을 끊을 수 있다.
- 트레이드오프: 레포 실행 전 `pip install -r requirements.txt`가 한 번 더 필요하고, CLI가 설치되지 않은 환경에서는 기존보다 명시적으로 의존성 오류가 드러난다.

## 리스크 및 미해결 이슈

- `twitter-cli 0.8.5` 자체가 X 응답 변화에 취약할 수 있다. 이 문서는 업스트림 자동 업데이트 문제를 줄이는 것이지, 외부 API 변화 자체를 없애지는 못한다.
- 현재 fixture는 `0.8.5` structured output을 전제로 한다. 이후 버전 업그레이드 시 fixture와 adapter 계약을 함께 재검증해야 한다.
