# 전체크롤링-이미지보강 수행계획

> 검수 대상 계획서입니다. 범위에서 `update` 중단기준 변경은 제외합니다.

## 목표

`전체 크롤링`을 데이터 손실 없이 누락 점검/복구에 사용할 수 있게 만들고, `이미지 보강`을 `local_images` 재연결/다운로드와 `media` 재수집으로 분리한다.

## 현재 코드 확인 결과

- `web_viewer/script.js`는 전체 크롤링 버튼에서 `executeScrap('all')`을 호출한다.
- `server.py`는 `/api/run-scrap`에서 `total_scrap.py --mode all`을 실행한다.
- `total_scrap.py`는 `--mode all`을 `thread_scrap.py`, `twitter_scrap.py`, `linkedin_scrap.py`에 그대로 전달한다.
- `thread_scrap.py`의 `all`은 최신 simple DB를 로드한 뒤 새 수집 결과와 기존 항목을 병합한다.
- `twitter_scrap.py`의 `all`은 기존 full DB를 `all_posts_map`에 싣고 새 수집 결과와 병합한다.
- `linkedin_scrap.py`의 `all`은 `final_posts = self.posts`로 처리되어 기존 full DB를 보존 병합하지 않는다.
- `total_scrap.py::download_images()`는 `media`가 이미 있는 글만 로컬로 다운로드하고 `local_images`를 채운다.
- `utils/backfill_local_images.mjs`는 이미 존재하는 로컬 이미지 파일을 JSON의 `local_images`에 되살리는 도구이며, 원격 이미지를 새로 다운로드하거나 `media: []`를 채우지 않는다.

## 범위

포함:

- 전체 크롤링 실행 전후 진단 리포트
- LinkedIn `all` 모드의 기존 데이터 보존 병합
- 전체 크롤링 결과의 ID 감소 방지 검증
- `media`가 있는 글의 `local_images` 보강
- `media: []`인 기존 글의 재조회 대상 목록 산출

제외:

- `update` 모드 중단기준 변경
- 저장시간(`platform_saved_at`) 신규 스키마 도입
- UI 디자인 변경
- 태그/즐겨찾기 데이터 마이그레이션

## 영구화 Surface

- `output_threads/python/threads_py_simple_YYYYMMDD.json`

- `scrap_failures_twitter.json`

## 마이그레이션 판단

스키마 마이그레이션은 불필요하다.

근거:

- 기존 필드인 `media`, `local_images`, `sequence_id`, `platform_sequence_id`, `is_detail_collected`만 사용한다.
- JSON 구조를 바꾸지 않고 병합/검증/다운로드 흐름만 분리한다.
- 기존 산출물의 일괄 변환은 하지 않고, 새 실행 결과와 명시적 `--apply` 명령에서만 파일을 갱신한다.

## 작업 단계

### 1. 전체 크롤링 사전 진단 스크립트 추가

파일:

- 생성: `scripts/audit_full_crawl_state.mjs`
- 테스트: `tests/unit/test_audit_full_crawl_state.py`

동작:

- 최신 플랫폼별 full 파일과 최신 total 파일을 읽는다.
- 플랫폼별 `total`, `id_dupes`, `url_dupes`, `media_empty`, `media_posts`, `missing_local_image_posts`를 출력한다.
- `total_full`과 플랫폼 full의 ID 집합 차이를 출력한다.
- 기본은 읽기 전용이다.

검증:

```powershell
pytest tests/unit/test_audit_full_crawl_state.py -q
node scripts/audit_full_crawl_state.mjs --json
```

기대:

- 테스트 통과
- JSON 출력에 `threads`, `linkedin`, `x`, `total` 키가 포함됨

### 2. LinkedIn `all` 모드 보존 병합 적용

파일:

- 수정: `linkedin_scrap.py`
- 테스트: `tests/unit/test_linkedin_full_merge.py`

동작:

- `CRAWL_MODE == "all"`에서도 기존 full DB를 버리지 않는다.
- 새로 수집된 글은 기존 ID에 없을 때만 신규로 추가한다.
- 기존 ID가 다시 수집되면 `sequence_id`, `crawled_at`, 기존 `local_images`를 보존하고, 새 `media`/본문이 더 풍부할 때만 갱신한다.
- `all` 결과의 기존 ID 개수가 이전 full보다 감소하면 저장 전에 실패시킨다.

검증:

```powershell
pytest tests/unit/test_linkedin_full_merge.py -q
python -m py_compile linkedin_scrap.py
```

기대:

- 기존 3건 + 신규 1건 fixture에서 결과 4건
- 기존 `local_images`가 보존됨
- 부분 수집 fixture에서 기존 ID 감소 방지 예외 발생

### 3. 전체 크롤링 실행 전후 비교 리포트 추가

파일:

- 수정: `total_scrap.py`
- 수정: `server.py`
- 테스트: `tests/unit/test_total_scrap_full_audit.py`

동작:

- `mode == "all"`일 때 실행 전 최신 total/platform ID 집합을 기록한다.
- 실행 후 `added`, `retained`, `missing_after`, `media_empty_delta`, `local_image_missing_delta`를 계산한다.
- `missing_after`가 있으면 결과는 성공이어도 사용자 메시지에 “확인 필요” 상태를 포함한다.
- 파일 삭제나 자동 롤백은 하지 않는다.

검증:

```powershell
pytest tests/unit/test_total_scrap_full_audit.py -q
pytest tests/integration/test_run_scrap_stats.py -q
```

기대:

- `mode=all` 결과에 `full_audit` 객체가 포함됨
- `missing_after` fixture가 있으면 “확인 필요” 플래그가 켜짐

### 4. `local_images` 보강 명령 분리

파일:

- 수정: `total_scrap.py`
- 수정: `utils/backfill_local_images.mjs`
- 테스트: `tests/unit/test_backfill_local_images.py`

동작:

- `media`가 있는 글의 로컬 이미지 보강을 별도 명령으로 실행한다.
- `--dry-run`은 변경하지 않고 대상 건수와 실패 예상 URL을 보여준다.
- `--apply`는 백업 파일을 만든 뒤 `local_images`를 갱신한다.
- 새 원격 다운로드와 기존 파일 재연결을 결과에서 구분한다.

검증:

```powershell
pytest tests/unit/test_backfill_local_images.py -q
node utils/backfill_local_images.mjs --target output_total/total_full_20260507.json --dry-run
```

기대:

- dry-run은 파일을 변경하지 않음
- apply는 백업 파일을 만들고 `local_images`만 갱신

### 5. `media: []` 재조회 대상 목록 산출

파일:

- 생성: `scripts/audit_media_empty_targets.mjs`
- 테스트: `tests/unit/test_audit_media_empty_targets.py`

동작:

- 최신 total 파일에서 `media`가 빈 글을 플랫폼별로 집계한다.
- 본문만 있는 글과 이미지 누락 의심 글을 단정하지 않고 별도 그룹으로 표시한다.
- `platform`, `platform_id`, `url`, `created_at`, `full_text_preview`를 포함한 후보 목록을 JSON으로 출력한다.
- 실제 재조회/수정은 하지 않는다.

검증:

```powershell
pytest tests/unit/test_audit_media_empty_targets.py -q
node scripts/audit_media_empty_targets.mjs --platform linkedin --limit 20
```

기대:

- 후보 목록이 출력됨
- 파일 변경 없음

## 실행 순서

1. 사전 진단 스크립트부터 추가한다.
2. LinkedIn `all` 보존 병합을 먼저 고친다.
3. 전체 크롤링 후 비교 리포트를 붙인다.
4. `local_images` 보강 명령을 분리한다.
5. `media: []` 재조회 대상 목록만 산출한다.
6. 실제 전체 크롤링은 위 검증이 통과한 뒤 수동으로 1회 실행한다.

## 통합 검증

```powershell
python -m py_compile total_scrap.py thread_scrap.py linkedin_scrap.py twitter_scrap.py thread_scrap_single.py twitter_scrap_single.py
pytest tests/unit/test_backfill_local_images.py -q
pytest tests/contract/test_schemas.py -q
node utils/query-sns.mjs --help
node scripts/audit_full_crawl_state.mjs --json
```

현재 `tests/unit/test_total_scrap_orchestration.py`는 Windows `cp949` 디코딩 문제로 실패한다. 전체 작업 전에 해당 테스트의 subprocess 캡처 인코딩을 고정하거나, 별도 수정 범위로 분리한다.

## 완료 기준

- `전체 크롤링` 실행 후 기존 ID 감소가 보고되면 “확인 필요”로 표시된다.
- LinkedIn `all` 모드가 기존 full DB를 보존 병합한다.
- `media`가 있는 글의 `local_images` 보강은 별도 dry-run/apply로 실행된다.
- `media: []`인 글은 별도 후보 목록으로만 산출되고, update/all 흐름에 섞이지 않는다.
- 중단기준 코드는 이 계획에서 변경하지 않는다.
