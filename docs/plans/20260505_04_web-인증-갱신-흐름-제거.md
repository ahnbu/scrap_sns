## 의사결정 — 왜 길 A(제거)인가

### 비교

| 길 | 방식 | 장점 | 단점 | 채택 |
|---|---|---|---|---|
| A. 죽은 코드 제거 | `--web` 흐름 + 백엔드 API + 패널 UI 모두 제거 | 코드베이스 솔직, 안내문구 단순화, AI 안내 혼란 제거 | 코드 변경 범위 큼(약 500~700줄 제거) | ✅ |
| B. 보존 + 안내문구만 정비 | 코드는 그대로, README와 앱 안내만 "터미널이 디폴트" | 변경 단위 작음 | 죽은 코드가 계속 사람·AI를 혼란시킴 | ❌ |
| C. 제거 + 새 web 흐름 재설계 | 죽은 흐름 제거 후 v3 PoC 패턴으로 새 web 갱신 구축 | 사용자 UX 개선 | 변경 범위 매우 큼, smoke 테스트 별도 plan과 얽힘 | ⚠️ 별도 plan |

**A 채택 근거:**
- 사용자가 직접 확인: "인증 패널은 동작하지 않았다"
- 본 세션 v3/v4/v5 PoC가 터미널 갱신은 통과 확정 — 운영 경로는 이미 명확
- 죽은 코드 보존은 부채. AI가 안내문구 작성 시 잘못된 흐름을 추천하게 만듦
- 새 web 흐름은 별도 plan(`_05` 예정)에서 v3 PoC 패턴으로 재설계 가능. 그 전에 죽은 코드를 제거해야 깔끔
