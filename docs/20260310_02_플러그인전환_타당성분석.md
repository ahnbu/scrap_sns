### 3. my-session-dashboard 플러그인 구조 분석

#### 디렉토리 구조

```
my-session-dashboard/
├── .claude-plugin/plugin.json   # 최소 메타데이터 (name, version, description)
├── commands/ss.md               # /ss 슬래시 커맨드 (진입점)
├── hooks/hooks.json             # SessionStart 훅 선언
├── hooks/ensure-commands.js     # 커맨드 자동 등록 스크립트
├── skills/session-dashboard/SKILL.md  # 실행 흐름 기술 (Triggers, 단계별 지시)
├── build.js                     # JSONL → HTML 변환 빌더
└── serve.js                     # 로컬 HTTP 서버 + SSE + 브라우저 열기
```

#### 실행 흐름 (Command → Skill → Script)

```
/ss 입력
  → commands/ss.md (allowed-tools: Bash)
    → skills/session-dashboard/SKILL.md 참조
      → node serve.js 실행 (수초 내 완료)
        → 브라우저 자동 열기 + SSE로 진행률 전송
        → 5초 후 서버 자동 종료
```

**핵심 패턴**: Convention-based 자동 발견 (`commands/`, `skills/`, `hooks/` 폴더명), `plugin:` 마커로 소유권 추적

### 4. 핵심 기술 질문 검토

#### Q1. MCP(LinkedIn/Threads)로 스크래퍼를 대체할 수 있나?

**결론: 불가**. LinkedIn MCP / Threads MCP는 **본인 게시글** 조회 중심. 사용자의 핵심 니즈인 "북마크한 타인 게시글 수집"은 미지원.

#### Q2. Playwright CLI를 scrap_sns 내부에서 활용할 수 있나?

| 현재 Playwright 고급 기능 | CLI 지원 | 적용 범위 |
|---|---|---|
| `page.on("response")` 네트워크 가로채기 | ❌ 불가 | thread_scrap, linkedin_scrap, twitter_scrap (핵심 3개) |
| `launch_persistent_context()` | ❌ 불가 | twitter_scrap, twitter_scrap_single |
| `async_api` + `Semaphore(3)` 병렬 탭 | ❌ 불가 | thread_scrap_single |
| `headless=False` + 화면 밖 배치 (봇 우회) | ⚠️ 제한적 | 전체 스크래퍼 |
| 개별 URL 방문 + DOM 파싱 (Consumer) | ⚠️ 이론적 가능 | thread_scrap_single, twitter_scrap_single |

**결론**: 핵심 Producer 스크래퍼(네트워크 가로채기 기반)는 CLI 대체 불가. Consumer는 이론적 가능하나 async 병렬 탭이 불가하여 성능 저하. **Python Playwright async 병렬도 튜닝이 더 실용적**.

`sns-scrap` 스킬이 playwright-cli를 쓰는 것은 "개별 URL 1개 열어서 스냅샷"이라는 다른 use case.

#### Q3. Zero Rebuild vs 리팩터링 중 무엇이 나은가?

| 항목 | 리팩터링 | Zero Rebuild |
|------|----------|-------------|
| **스크래핑 노하우 보존** | ✅ 동작 코드 그대로 유지 | ❌ spec 옮기며 누락 위험 (anti-bot 트릭, 셀렉터 폴백 등) |
| **작업량** | ████░░ 공통모듈 추출 + 중복 제거 | ██░░░░ PRD 추출 + 전체 재구현 + 검증 |
| **리스크** | ✅ 점진적, 각 단계 검증 가능 | ❌ "동작하던 것이 안 되는" 리그레션 가능 |
| **결과물 깔끔함** | ██░░░░ 유기적 성장 흔적 일부 잔존 | ✅ ██████ 처음부터 일관된 구조 |
| **모듈화 달성도** | ████░░ 중복 제거 + 인터페이스 통일 | ██████ 설계부터 분리 |
| **소요 시간** | ✅ ████░░ 단기 (핵심 6중 중복 제거 중심) | ░░░░░░ 장기 (스크래핑 재검증 포함) |
| **병행 가능성** | ✅ 기존 코드 동작하면서 점진 수정 | ❌ 새 코드 완성까지 구 코드 유지 필요 |
| **현재 에러 해결 속도** | ✅ 즉시 수정 가능 | ❌ 재구현 완료 후에야 해결 |

**결론**: 이 프로젝트의 가치는 "SNS별 anti-bot 우회 노하우가 동작하는 코드"이므로 **리팩터링 우세**.

## 리스크 및 미해결 이슈

### ⚠️ Blind Spot 1: Playwright anti-bot 의존성 악화 위험
Threads/LinkedIn이 봇 탐지를 강화하면 스크래퍼 전체가 무력화됨. 플러그인화 투자가 매몰 비용이 될 수 있음.
→ 완화책: 수집 실패율을 smoke 테스트로 모니터링. 2회 연속 세션 완전 차단 시 대안 탐색 트리거.

### ⚠️ Blind Spot 2: 플러그인 경로 잠금 문제
플러그인은 `~/.claude/my-claude-plugins/`에 위치해야 하지만, 현재 `D:\vibe-coding\scrap_sns`의 output 디렉토리·인증 파일·웹 뷰어 데이터 경로 관리가 복잡해짐.
→ 완화책: symlink 또는 설정 파일(config.json)로 데이터 경로 외부화.

### 미해결: Playwright CLI Consumer 전환 실효성
타래글·X 타래글 개별 URL 수집 시 Playwright CLI 부분 적용 가능성은 이론적으로 존재하나 검증 안 됨.

## 핵심 결론

### 1. 플러그인 전환: 조건부 가치 있음 (현재는 시기상조)
- 스크래퍼는 수분~수십분 실행 + GUI 브라우저 + 대화형 로그인 → `/ss`처럼 즉각 결과를 보여주는 패턴과 맞지 않음
- 기존 프로젝트가 안정적으로 돌아가야 래핑의 의미가 있음
- **순서: 리팩터링 → 스킬 래핑 → (선택) 플러그인화**

### 2. Playwright CLI 전환: 핵심 스크래퍼는 불가
- Producer(thread_scrap, linkedin_scrap, twitter_scrap): `page.on("response")` 네트워크 가로채기 필수 → CLI 대체 불가
- Consumer(thread_scrap_single, twitter_scrap_single): 개별 URL 방문이라 이론적 가능하나, async 병렬 탭·persistent_context 등 고급 기능 의존
- 결론: Python Playwright async의 병렬도 튜닝이 더 실용적

### 3. MCP 대체: 제한적
- LinkedIn MCP / Threads MCP는 본인 게시글 조회용
- 사용자 니즈(북마크한 타인 게시글 수집)와 불일치

### 4. Zero Rebuild vs 리팩터링: 리팩터링 우세
- 프로젝트 핵심 가치 = "SNS별 anti-bot 우회 노하우가 동작하는 코드"
- Zero Rebuild는 spec 이전 시 노하우 누락 리스크가 높음
- 리팩터링은 점진적·검증 가능·병행 가능
