## 🔴 크리티컬 결함·리스크

### 1. **LinkedIn `all` "기존 ID 감소 시 실패시킨다"는 정상 케이스에서 오작동 위험**
plan 2단계 명세: *"`all` 결과의 기존 ID 개수가 이전 full보다 감소하면 저장 전에 실패시킨다."*

- LinkedIn 저장 글은 **사용자가 의도적으로 unsave 가능**하다. 사용자가 SNS에서 저장 해제한 글은 다음 `all` 수집에서 정당하게 사라진다.
- 이걸 "실패"로 처리하면 **정상 사용 케이스를 막는다**. plan 3단계의 "`missing_after` → 확인 필요 플래그"와도 모순(같은 현상을 한쪽은 fatal, 한쪽은 warning).
- **권고**: 2단계도 fatal stop 대신 "기존 데이터를 base로 깔고 신규 수집을 머지"만 하라. 감소 보고는 3단계 audit으로 통합. 또는 "감소량이 임계치(예: 5%)를 넘을 때만 차단" 같은 합리적 가드.

### 2. **Step 3 "missing_after" 정의 불명확 — 거짓 양성 폭발 위험**
LinkedIn `all`은 페이지를 끝까지 스크롤해야 모든 ID를 본다. 페이지 무한스크롤이 중간에 끊기면(네트워크 오류, 인증 만료, 봇 탐지) 부분 수집 결과가 나오고, 그러면 "missing_after"가 매번 대량 발생할 가능성이 있다. 사용자는 "확인 필요" 플래그에 둔감해질 것이다.

LinkedIn의 부분 수집 vs unsave 구별 메커니즘이 plan에 없다. 최소한 **수집된 raw 페이지/응답 횟수**나 **중단 사유(끝까지 스크롤됨/타임아웃/오류)** 메타가 같이 비교되어야 한다.

### 3. **Step 4 명세 충돌: 명령 분리 vs 기존 download_images 유지**
- plan은 "별도 명령으로 실행한다"라고 하면서, 기존 `total_scrap.py::download_images()`가 자동 호출되는 부분(`total_scrap.py:555`)을 어떻게 처리할지 침묵한다.
- 그대로 두면 기존 자동 다운로드 + 신규 별도 명령이 **중복 실행**되어 "별도 분리"의 의미가 없다.
- 또한 plan의 "새 원격 다운로드와 기존 파일 재연결을 결과에서 구분한다"는 mjs 확장은 **fetch + 재시도 + UA 분기(licdn 등)**를 mjs에 추가해야 하는데, 검증 명령에는 dry-run만 있어 원격 다운로드 동작이 검증 누락이다.
- **이미 `total_scrap.py::download_images`가 `linked_existing` vs `count`(신규 다운로드)를 print로 구분**하고 있다. 별도 분리 vs 기존 강화 중 무엇이 목적인지 결정 필요.

### 4. **Step 5의 "본문만 있는 글 vs 이미지 누락 의심 글" 분류 기준이 비어 있음**
*"본문만 있는 글과 이미지 누락 의심 글을 단정하지 않고 별도 그룹으로 표시한다"* — 어떻게 판별?
- Threads/X는 텍스트만 있는 포스트가 다수다(정상).
- LinkedIn은 글만 있는 게시물이 정상.
- 분류 휴리스틱이 없으면 결과는 사실상 "전체 media empty 목록" 한 덩어리가 된다 (그룹 분리의 가치 없음).
- 기준: 예) "동일 작성자의 다른 글 90% 이상에 미디어가 있다" / "본문에 이미지/영상 키워드/imgurl 패턴이 있다" 등이 명세되어야 의미가 있음.

## 🟡 과최적화 신호

1. **새 스크립트 4개·테스트 4개·도구 1개 수정 = 합 9개 surface**가 추가되는데, 핵심 문제는 사실상 **하나** ("LinkedIn `all` 보존 병합 누락"). 나머지(audit·분리 명령·후보 산출)는 부수 인프라.
2. Step 5 "후보 목록 산출"은 실 액션 없는 보고서일 뿐. **현재 `media: []` 글이 몇 건이 있고 사용자가 무엇을 할지** 확인 후 만들어도 늦지 않다.
3. `tests/unit/test_total_scrap_orchestration.py` cp949 실패는 plan과 별개 이슈인데 Plan 검증 통과의 차단 요소처럼 끼워져 있다. 별도 처리하라는 단서가 있긴 하나 작업 시작 전 의존을 만든다.

## 권고 우선순위

1. **Step 2만 우선 실행** (LinkedIn 보존 병합) — 단, "감소 시 fatal 실패"는 fatal 대신 warning으로. 이게 plan의 80% 가치.
2. Step 1·3은 **하나로 통합**. 실행 전후 audit JSON 1개 + warning 메시지로 충분. 거짓 양성 줄이는 최소 휴리스틱 포함.
3. Step 4는 **결정 보류**. 사용자가 실제로 쓸 케이스(매번 보강 vs 1회성 마이그레이션)를 먼저 확정.
4. Step 5는 **실제 `media: []` 분포를 한 번 출력해보고 결정**. 분류 기준이 안 나오면 단순 카운트로 충분.
