# 검수 결과 — X 인증 오탐·프롬프트 복구 plan

## 한 줄 결론
**목적은 대체로 달성 가능하나, 두 개의 크리티컬 결함(Chromium user_data lock 충돌, server 재검증 동기 호출의 30초 블로킹)을 해결하지 않으면 같은 사용자 경험에서 새로운 장애가 발생한다.**

---

## 🔴 크리티컬 결함

### C1. `verify_x_auth.py`와 X producer의 user_data lock 충돌 위험
- `verify_x_auth.py:51`은 `user_data_dir = x_user_data()`를 그대로 사용해 `launch_persistent_context`를 띄운다.
- `twitter_scrap.py:254-264`의 X producer도 같은 디렉터리를 사용한다.
- Chromium은 동일 user_data_dir 동시 launch 시 `SingletonLock`/`ProfileLock`으로 실패한다.
- plan Task 3은 producer subprocess가 완전히 종료된 뒤 재검증을 돌리는 흐름을 전제하지만, **producer가 비정상 종료(`exit code 86` + 컨텍스트 close)했을 때 lock 파일이 깨끗하게 풀리는지 검증 명시가 없다**. 특히 transient launch 재시도 로직(Task 2)을 추가한 직후라면 stale lock 가능성이 더 높다.
- 결과: 오탐을 잡으려고 추가한 재검증이 lock 충돌로 실패 → `auth_recheck.reason = "verify_timeout"` → `auth_required`가 유지 → **사용자가 보는 증상은 전혀 개선되지 않는다.**
- 대응: (1) 재검증 전에 명시적 sleep + lock 파일(`SingletonLock`, `SingletonCookie`) 존재 체크, 또는 (2) `verify_x_auth.py`에 `--copy-profile` 모드를 추가해 임시 디렉터리로 복제 후 검증, 또는 (3) `subprocess.run` 호출 시 lock 충돌 에러 메시지를 별도 `reason`으로 분류.

### C2. `/api/run-scrap` 응답 지연 30s가 사용자 UX를 악화
- Task 3 Step 3의 `timeout=30`은 producer + consumer 둘 다 headful Chromium launch를 거치므로 실제로는 5~25초가 늘 추가된다.
- `run_scrap()`은 이미 X/Threads/LinkedIn 직렬 수집으로 분 단위 응답이라 큰 영향 없다고 볼 수 있으나, **이 재검증은 producer/consumer Chromium 창을 다시 띄운다** — 사용자 화면에 또 다른 창이 깜빡인다(headful 가능성). `verify_x_auth.py`는 headless=True지만, "X auth가 잘못 발화된 후 재검증 시 또 헤드리스 창 launch → lock 충돌"의 사이드이펙트를 plan이 인지하지 않았다.
- 대응: 리스크 표에 "30초 느려질 수 있다"만 한 줄 적혀 있을 뿐 사용자 시각 영향(Chromium 깜빡임)이 없음. 명시 필요.

## 🟠 중대 결함

### M1. Task 5 Step 2의 Expected가 비논리적
- 라인 800–812: 같은 테스트 코드를 `Step 2`에서 "PASS"로 예상하고, `Step 3`에서 "실패할 경우" 수정한다는 분기.
- TDD 흐름인 다른 Task(Task 1/2/3/4: 먼저 실패 → 구현 → 통과)와 일관성이 깨졌다.
- 더 중요한 문제: 현재 `getAuthRequiredPlatforms()` (script.js:375-381)는 `platform_results[x].auth_required` 또는 status에 `'auth'` 포함 시 platforms에 추가한다. **Task 3에서 server가 `status: "failed"`로 바꾸고 `auth_required`/`auth_required` 키도 제거한다면, 이미 통과한다**. 그런데 plan은 "통과하면 회귀 방지로 충분"이라는 가정에 그쳐, 실제 PASS 조건이 코드 흐름과 맞는지 plan 단계에서 확인을 미룬다. 검증 가능성 ↓.
- 대응: Step 1에서 직접 현재 코드를 시뮬레이션해서 PASS/FAIL 중 어느 쪽인지 확정하고, 명확한 한 갈래로 plan 작성.

### M2. Task 2 Step 4의 helper 시그니처 혼란
- 라인 346~379: 같은 `wait_for_bookmark_signal`을 두 가지 시그니처(`(page, state: dict)` vs `(page)`)로 연이어 정의하고 "실제 구현 시 …처럼 만든다"고 적었다. 구현자가 어느 쪽을 채택해야 하는지 모호하고, plan-check 시 "어떤 코드가 적용됐는지" 판별이 어렵다.
- 대응: 최종 시그니처 하나만 plan에 남기고 다른 한 버전은 삭제.

### M3. Task 3 Step 3의 `subprocess.run` 예외 처리 누락
- 라인 510-518에 `completed = subprocess.run(...)`를 적고, 라인 537-541에 별도 try/except로 `TimeoutExpired`를 잡는 코드가 또 등장. 두 블록이 어떻게 합쳐지는지 명시 안 됨.
- `FileNotFoundError`(Python 실행 실패), `OSError` 등 다른 예외도 catch 못 함 → server crash.
- 대응: 최종 helper 본문을 하나로 통합 제시.

### M4. Task 3 테스트의 `subprocess.run` mock이 과도하게 좁다
- 라인 472-474: `patch("subprocess.run", return_value=probe)`는 server.py 모듈의 모든 `subprocess.run`을 가린다. 다른 테스트가 같은 모듈에서 `subprocess.run`을 호출하면 광역 mock으로 인해 의도치 않은 영향이 생길 수 있다. (이 테스트 한 함수 내에서는 OK지만, 다른 라인의 `subprocess.run`이 호출된다면 다 같은 mock 응답을 받는다.)
- 또한 `_run_x_auth_recheck`는 `script_path` 존재 체크를 하므로, 테스트 환경에서 실제 파일이 있어야 한다. tmp 환경에서 PROJECT_ROOT의 실제 파일을 그대로 두는지 확실하게 명시 필요.

## 과최적화 의심

- **Task 3의 server측 재검증**: producer 판정 로직(Task 1/2)이 개선되면 오탐 자체가 사라지는 것이 정상 경로다. 그런데 plan은 producer 수정 + server 재검증을 둘 다 도입한다. 이중 안전망의 가치는 있으나, **재검증 호출이 새로 도입한 lock 충돌 리스크(C1) > 잡으려는 오탐 잔재**라면 순익이 음수일 수 있다.
- 대응 권장: producer 측 수정(Task 1·2)을 먼저 배포·관찰하고, 잔존 오탐이 실측되면 Task 3 도입. plan-check 검수 시점에선 "이미 한 PR에 묶었다"가 우선이라면 최소한 C1 대응이 plan 안에 명시되어야 한다.
