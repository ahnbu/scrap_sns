## ⚠️ 크리티컬 리스크 및 트레이드오프

이 계획은 구조를 개편하지 않고 국소적 패치로 안정성을 높이지만, OS 레벨의 포커스 제어에 의존하기 때문에 발생하는 명확한 트레이드오프가 있습니다.

*   **크리티컬 리스크 (Focus Stealing):** 
    재시도 루프(최대 3회, 1초 간격) 내내 `prepare_owned_chrome_window_for_bind(owned_hwnd)`가 호출됩니다. 만약 사용자가 다른 작업을 하고 있을 때 수집이 백그라운드에서 돌게 되면, **1초 간격으로 최대 3번 연속 사용자 창의 포커스를 강탈**하게 됩니다. 이는 Vibe-coding 시 매우 큰 방해 요소입니다.
*   **트레이드오프:**
    *   **얻는 것:** 기존 인프라(Chrome 프로필, OpenCLI)를 유지하며 wrong-bind에 의한 데이터 오염 및 사용자 탭(GitHub 등) 강제 종료 버그 완벽 차단.
    *   **잃는 것:** 수집 시 일시적인 화면 깜빡임과 사용자 입력 끊김 현상 심화. (OpenCLI가 특정 HWND를 타겟하지 못하는 이상 불가피한 선택)

## ✅ 검증 기준 평가 및 기타

*   **목적 vs 기준 충돌:** 없음. wrong-bind 시 `unbind`만 하고 `close`하지 않겠다는 목적이 Task 1의 테스트 케이스와 Task 2/3의 구현 지침에 완벽하게 일치합니다.
*   **완료 기준의 충분성:** 각 Task의 테스트 명령어와 기대 출력이 매우 구체적으로 작성되어 있어, 추후 `done-check`가 로그 기반으로 확정적인 판단을 내릴 수 있습니다.
*   **과최적화:** 발견되지 않음. 파이프라인 전체를 뜯어고치지 않고, 문제의 병목(active window bind)만 최소한으로 수정한 훌륭한 접근입니다.
