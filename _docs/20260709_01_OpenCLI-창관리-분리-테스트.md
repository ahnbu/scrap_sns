# OpenCLI 창관리 분리 테스트

## 1. 배경

SNS Scrap LinkedIn 수집 후 Chrome 상단에 OpenCLI 디버깅 배너가 남는 문제가 있었다.

기존 방식은 OpenCLI가 창 열기, 수집, 정리까지 모두 담당하는 구조였다. 그러나 `opencli browser close`는 실제 Chrome 창 닫기가 아니라 browser session tab lease 해제에 가깝다. 이 때문에 OpenCLI에 창 수명 관리를 맡기면 사용자 Chrome 창에 LinkedIn 탭이나 디버깅 배너가 남을 수 있다.

## 2. 가설

창 수명 관리와 수집 역할을 분리하면 문제가 단순해진다.

- Chrome/Windows가 수집 전용 창 생성과 종료를 담당한다.
- OpenCLI는 이미 열린 수집용 창에 `bind`해서 페이지 상태, DOM, network, eval 수집만 담당한다.
- 수집 완료 후 저장한 window handle로 수집용 창만 닫으면 사용자 SNS Scrap 창을 오염시키지 않을 수 있다.

## 3. 테스트 전 기준선

엄격한 클린 상태를 먼저 만들었다.

```markdown
OpenCLI daemon: not running

OpenCLI session tab list:
- linkedin_saved_production: []
- linkedin_saved_shadow: []
- persistent: []

Chrome 창:
- SNS Feed Viewer - Chrome
- Hwnd: 0x4C08A4
```

## 4. 테스트 절차

1. Chrome `--new-window`로 테스트 창을 생성한다.
2. Win32 API로 테스트 창의 `Hwnd`를 식별한다.
3. 테스트 창을 전면화한 뒤 `opencli browser <session> bind`를 실행한다.
4. OpenCLI `state`, `tab list`로 테스트 URL과 DOM을 확인한다.
5. `unbind -> close -> daemon stop`을 실행한다.
6. 저장한 `Hwnd`로 테스트 창에 `WM_CLOSE`를 보낸다.
7. 최종 Chrome 창 목록과 OpenCLI daemon 상태를 확인한다.

## 6. 판단

축소 테스트 기준으로 역할 분리 구조는 동작했다.

```markdown
Chrome/Windows: 수집용 창 생성과 종료 담당
OpenCLI: 이미 열린 수집용 창에 bind해서 읽기만 담당
```

이 구조는 OpenCLI에 창 생성/종료까지 맡기는 기존 방식보다 안정적이다. 특히 `opencli browser close`가 실제 창 닫기가 아닌 상황에서, 창 종료는 window handle 기준으로 처리하는 편이 더 정확하다.

## 7. 다음 적용 방향

LinkedIn 수집에 적용할 때는 다음 방향이 적절하다.

1. LinkedIn saved-posts URL을 수집 전용 Chrome 새 창으로 연다.
2. 열린 창의 window handle을 저장한다.
3. OpenCLI는 해당 창에 `bind`한다.
4. 기존 수집 로직은 OpenCLI `state`, `network`, `eval`, `scroll`을 사용한다.
5. 수집 완료 또는 실패 시 OpenCLI `unbind -> close -> daemon stop`을 실행한다.
6. 저장한 window handle 기준으로 수집용 창만 닫는다.
7. 사용자 `SNS Feed Viewer - Chrome` 창은 닫거나 전환하지 않는다.

## 8. 남은 검증

- 실제 LinkedIn saved-posts URL에서 같은 구조가 동작하는지 확인한다.
- `whoami --site-session persistent`를 제거하거나, 수집용 창 안에서 로그인 상태를 확인하는 방식으로 대체한다.
- 실제 SNS Scrap update 흐름에서 사용자 Chrome 창에 OpenCLI 디버깅 배너가 남지 않는지 확인한다.
- 수집 실패 경로에서도 수집용 창과 OpenCLI daemon이 정리되는지 확인한다.

### 9.1 테스트 목적

앞선 `example.com` 테스트는 창 수명주기만 검증했다. 이번 추가 검증은 다음을 확인하기 위한 것이다.

- 실제 LinkedIn saved-posts URL도 Chrome/Windows가 연 창에 OpenCLI가 `bind`할 수 있는가.
- `whoami --site-session persistent` 없이 bound 창 안에서 saved-posts 로그인/페이지 상태를 확인할 수 있는가.

- cleanup 후 수집용 LinkedIn 창만 닫히고 `SNS Feed Viewer - Chrome` 창이 유지되는가.

### 9.2 테스트를 위한 collector 옵션

`scripts/linkedin_opencli_shadow_collect.mjs`에 테스트 옵션을 추가했다.

```markdown

```

이 옵션이 있으면 collector는 내부에서 다음 명령을 실행하지 않는다.

```javascript
browser(session, ["open", url, "--window", "background"], { expectJson: true });
```

대신 이미 bind된 session을 그대로 사용한다. 옵션이 없으면 기존 동작은 유지된다.

### 9.5 판단

실제 LinkedIn saved-posts에서도 역할 분리 구조가 동작했다.

```markdown
Chrome/Windows:
- saved-posts 수집용 창 생성
- window handle 저장
- 수집 후 해당 창만 닫기

OpenCLI:
- 이미 열린 LinkedIn 창에 bind
- bound session에서 state/network/eval 기반 raw 수집
- unbind/close/daemon stop cleanup
```

중요한 점은 `whoami --site-session persistent` 없이도 bound 창의 title이 `저장한 게시물 | LinkedIn`으로 확인됐고, collector가 raw 수집에 성공했다는 것이다. 따라서 로그인 확인은 별도 `whoami` 명령보다 수집용 창 안에서 `state` 또는 `eval`로 대체하는 방향이 가능하다.

### 9.6 남은 검증 업데이트

- 실제 SNS Scrap update 흐름에 이 구조를 통합했을 때도 사용자 Chrome 창에 디버깅 배너가 남지 않는지 확인해야 한다.
- 실패 경로에서도 `unbind -> close -> daemon stop -> Hwnd 닫기`가 항상 실행되는지 확인해야 한다.
- `--max-pages 1`에서 2페이지가 수집되는 현재 collector 흐름은 별도 개선 대상이다. 이번 테스트의 핵심 blocker는 아니다.
